## Rules

### Rust Error Handling Pattern
**Source**: `AGENTS.md` (Error Handling), codebase patterns in `crates/*/src/error.rs`
**Rule**: In Rust library code for this project:
- Always use `thiserror` for error enums — never `anyhow` (anyhow is only for binary entry points and examples)
- Every error enum must be `#[non_exhaustive]`
- Provide factory constructors accepting `impl Into<String>` for string-based variants
- Define `pub type <CrateName>Result<T> = Result<T, <CrateName>Error>` alias
- In tests, `.unwrap()` and `.expect()` are acceptable (`allow-unwrap-in-tests = true` in `clippy.toml`) — but in production code, document the reasoning if you must use them
- Propagate errors with `?`; avoid panicking in library code

### Rust Module and Naming Conventions
**Source**: `AGENTS.md` (Naming Conventions, Module Structure), `clippy.toml`
**Rule**: Nebula-specific conventions to follow and enforce:
- Crate names: `kebab-case` with `nebula-` prefix (`nebula-action`, `nebula-storage`)
- Action/plugin keys: `dot.notation.lowercase` (`http.request`, `math.add`) — never PascalCase or underscores
- Every crate exposes `pub mod prelude` with star-importable types
- Section dividers in `lib.rs` and large source files: `// ── Section Name ────────────────────────────────────────────────────`
- Functions exceeding 100 lines trigger clippy (`too-many-lines-threshold = 100`) — split proactively; avoid writing functions over 80 lines
- Builder methods must use consuming `self` (not `&mut self`) and be annotated `#[must_use]`
