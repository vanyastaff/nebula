## MEDIUM

### M-1 — `cargo deny` does not enforce layers (contract violation)
- **Where:** `deny.toml` `[bans]` section (lines 167+).
- **What:** `[bans.deny] = []` is empty. `multiple-versions = "warn"`. There
  are no layer bans or version bans. CI runs `cargo deny check` (via
  `.github/workflows/ci.yml`), but only advisories/licenses/sources fire.
- **Why it matters:** `CLAUDE.md` and (until this run) `.project/context/ROOT.md`
  both claimed `Layers enforced by cargo deny: Infra → Core → Business → Exec → API`.
  This is **aspirational, not enforced**. Any cross-layer `nebula-*` dep added
  by hand will pass CI silently.
- **Auto-fix applied:** `ROOT.md` and `pitfalls.md` now flag this honestly
  ("convention-only"). `CLAUDE.md` still has the stale claim — left for human
  review (touching top-level `CLAUDE.md` is a higher-blast-radius edit).
- **Manual action suggested:** either (a) add real `[[bans.deny]]` entries for
  forbidden dep edges, or (b) update `CLAUDE.md` to match `ROOT.md`.

## LOW

### L-1 — Duplicate dependency versions (ecosystem bloat)
`cargo tree --duplicates` lists 22 crate families with two versions pulled in:

- **RustCrypto split:** `block-buffer` (0.10/0.12), `digest` (0.10/0.11),
  `sha2` (0.10/0.11), `hmac` (0.12/0.13), `crypto-common` (0.1/0.2),
  `cpufeatures`, `const-oid`. Root cause: `argon2` + `jsonwebtoken` pull older
  versions, `nebula-credential` + `nebula-action` use newer.
- **rand split:** `rand` (pre-0.10/0.10), `rand_core`, `rand_chacha`, `getrandom`.
- **Windows crates:** `windows-sys`, `windows-targets`, `windows_x86_64_msvc` —
  typical Windows build duplication.
- **Proc-macro helpers:** `darling`, `darling_core`, `darling_macro` (version
  split between two macro-crate consumers).
- **Misc:** `hashbrown` (0.14/0.15), `foldhash`, `crossterm`, `itertools`.

- **Why it matters:** binary bloat, longer link times, potential type
  incompatibility surface if both versions of e.g. `digest::Digest` appear in
  public API (they don't today).
- **Manual action:** most require waiting for upstream bumps (argon2,
  jsonwebtoken). Short-term: bump deny.toml `multiple-versions` to "deny"
  with explicit `skip` for known-unfixable roots, to prevent *new* drift.

### L-2 — `tokio::spawn(drain_plugin_stderr)` handle dropped
- **Where:** `crates/sandbox/src/process.rs:321`.
- **What:** fire-and-forget task drains the plugin's stderr into `tracing`.
  Comment is explicit: "task ends when the child's stderr closes — usually on
  plugin exit". `child.kill_on_drop(true)` is set at line 284 so dropping the
  `ProcessSandbox` kills the child, which closes stderr, which ends the task.
  The chain is correct — but the invariant is *very* implicit.
- **Manual action suggested:** not a bug today. A comment on the spawn site
  stating "lifecycle bounded by `kill_on_drop` above → stderr EOF → task exit"
  would make the contract explicit for future readers.

### L-3 — Synchronous `std::fs` in `plugin-sdk/src/transport.rs::bind_unix`
- **Where:** `crates/plugin-sdk/src/transport.rs:62-68`.
- **What:** `bind_listener()` is a synchronous `fn` (not `async fn`) and calls
  `std::fs::{remove_dir_all, create_dir, set_permissions}`. This is fine *if*
  callers invoke it from a non-async entry point (plugin-side main, before
  tokio runtime starts).
- **Manual action:** verify all call sites. If any caller runs inside a
  tokio task, wrap the body in `tokio::task::spawn_blocking`. Current call
  graph was not traced to ground in this pass.
