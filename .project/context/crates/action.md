# nebula-action
Action trait hierarchy and execution contract. Defines **what** actions are; the
engine runs them from `nebula-runtime`. `ActionRegistry` lives there, not here.

## Subtypes
- `StatelessAction` — one-shot.
- `StatefulAction` — `Continue`/`Break` loop. `State: Serialize + DeserializeOwned + Clone`.
- `TriggerAction` — starts workflows from outside the graph. DX specializations: `WebhookAction` (push, returns `WebhookResponse`) + `PollAction` (pull, returns `PollCycle`).
- `ResourceAction` — branch-scoped DI, single `type Resource`.

## Dispatch
- `ActionHandler` (`#[non_exhaustive]`, `Clone`) — engine's sum type. Variants wrap `Arc<dyn XxxHandler>`.
- `{Stateless,Stateful,Trigger,Resource}Handler` — JSON-erased contracts. Typed authors write `impl XxxAction`; matching `XxxActionAdapter` bridges typed ⇄ JSON at registration.
- Canonical path: `nebula_action::{ActionHandler, StatelessHandler, IncomingEvent, ...}` from crate root. No `handler::X` alias surface.

## Credentials
- `CredentialGuard<S: Zeroize>` — only credential type re-exported here. Deref + Zeroize on drop, `!Serialize` (compile error in `Output`/`State`).
- `ctx.credential::<S>() -> CredentialGuard<S>` via `TypeId`. `ctx.credential_by_id(id)` for string-keyed access.
- `ActionDependencies` declares: `credential_keys`/`resource_keys` (typed, validated at registration) + `credential_types() -> Vec<TypeId>` (for `ScopedCredentialAccessor` sandboxing). Default empty.
- `#[derive(Action)]` + `#[action(credentials = [T1, T2])]` generates both. Duplicate types → compile error.

## Errors
- `ActionError::Validation { field: &'static str, reason: ValidationReason, detail: Option<String> }`. `field` is compile-time constant by type — cannot carry user input. `detail` is sanitized (control chars → `\uXXXX`, truncated to `MAX_VALIDATION_DETAIL = 256` B).
- `retryable(..)` vs `fatal(..)` decides retry. `ActionErrorExt` gives `.retryable()?`/`.fatal()?` on any `Result`.
- `RetryHintCode` (`#[non_exhaustive]`) — action-supplied retry hint. Attached via `retryable_with_hint`/`fatal_with_hint`. Distinct from `nebula_error::Classify::code()`.
- Payload is `Arc<anyhow::Error>` — full chain, cheap clone.
- `ActionError::CredentialRefreshFailed { action_key, source: Arc<dyn Error + Send + Sync> }` (#306, Batch 5D) — surfaced by the engine when the proactive `credential_refresh` hook fails before action dispatch. Default classification is **retryable** (transient credential-store outages); appears in `Classify::code()` as `ACTION:CREDENTIAL_REFRESH_FAILED` / `ErrorCategory::External`. `source` is `Arc<dyn Error>` (not `Box`) so the variant stays `Clone` — matching the pattern of the other `Arc<anyhow::Error>` payloads. Construct via `ActionError::credential_refresh_failed(action_key, source)`. The enum stays `#[non_exhaustive]`, so this is a purely additive change for downstream consumers.

## Traps
- **State checkpointing.** `StatefulActionAdapter::execute` MUST flush `typed_state` back to JSON on both Ok and Err paths before propagating. A `Retryable` with mutated cursor that isn't flushed makes the engine replay — duplicated API calls, double charges, double emits. Only exception: `Validation` from input/state deserialization (typed state never existed). If state serialization itself fails on the error path, log and propagate the original error — masking breaks retry classification.
- **Webhook double-start.** `WebhookTriggerAdapter::start` rejects double-start with `Fatal` (read-lock pre-check + write-lock re-check with orphan `on_deactivate`). Silent re-registration leaks external webhook registrations. Guards scoped so they never cross `.await` (parking_lot is `!Send`). W2 failure mode: if the orphan `on_deactivate` network call itself fails, the external hook stays live — operational debt tracked in `webhook-api-v2.md` as V3 reconcile loop.
- **Webhook in-flight tracking.** `handle_event` increments an `Arc<AtomicU32>` in-flight counter via `InFlightGuard` RAII drop that also calls `Notify::notify_waiters`. `stop()` parks on `Notify::notified()` until the counter hits zero (no `yield_now` busy spin). Prevents state destruction while requests are mid-flight AND doesn't burn a runtime task wake per cycle while waiting.
- **Webhook error path sends 500 via oneshot before propagating Err.** On `handle_request` failure, the adapter sends `WebhookHttpResponse::new(500, empty)` via the attached oneshot channel BEFORE returning the `ActionError` to the runtime. Without this, the transport saw `RecvError` and had to guess the status code. Same shape applies to the "state is None" path (before-start or after-stop race) and the cancellation path (sends 503 instead). Health `record_error()` fires on any of these.
- **Webhook handle_event is cancellation-safe.** `handle_request` is wrapped in `tokio::select!` against `ctx.cancellation`. On cancel mid-request: sends 503, records error, returns `ActionError::retryable("cancelled mid-request")`. `stop()` triggered mid-dispatch no longer blocks on a hung handler; the in-flight counter drops and `Notify` wakes the `stop()` waiter.
- **Webhook TriggerHealth parity with poll.** `record_success(n)` on `Emit(_)`/`EmitMany(v)` outcomes, `record_idle()` on `Skip`, `record_error()` on handler Err / cancellation / before-start state miss. No idle streak concept (push model).
- **HMAC primitives are constant-time, single-valued, replay-aware.** (1) `verify_hmac_sha256` rejects multi-valued signature headers via `headers.get_all().iter()` strict count — proxy-chain duplication attack defence. (2) Timing-invariant: hex decode and MAC computation always run regardless of header well-formedness; results are AND'd. (3) `verify_hmac_sha256_with_timestamp` handles Stripe / Slack / Mailgun replay-window schemes — takes a `canonicalize_fn` closure so one helper serves every `{ts}.{body}` or `v0:{ts}:{body}` variant, uses `request.received_at()` as clock source (never wall), rejects skew > tolerance and forward skew > 60s. (4) `verify_hmac_sha256_base64` for Shopify/Square (same primitives, base64 decode instead of hex). (5) Empty secret is always `Err(Validation)` — fail-closed.
- **`WebhookEndpointProvider` capability on `TriggerContext`.** Optional `ctx.webhook: Option<Arc<dyn WebhookEndpointProvider>>` field — `Some` for webhook triggers, `None` for poll/other shapes. Trait declared in `nebula-action::webhook`, implemented by `nebula-api::webhook::EndpointProviderImpl` (API → Business, no upward dep). Action reads `ctx.webhook.as_ref()?.endpoint_url()` inside `on_activate` to get the public URL it must register with the external provider. Before this capability, action authors had no way to discover the URL from `TriggerContext` — this was the #1 DX blocker flagged by the sdk-user review.
- **`body_json_bounded<T>(max_depth)` guards against JSON bomb.** Pre-scans body bytes with a brace/bracket depth counter (skipping string contents with escape handling) and rejects before invoking `serde_json` if nesting exceeds `max_depth`. Default recommendation 64. Plain `body_json` keeps its shape but docstring carries a `# Security — depth not checked` block pointing at the bounded variant. Tokio worker threads have 2 MiB stacks and real 100k-deep input overflows.
- **Poll double-start.** `PollTriggerAdapter::start` rejects via `AtomicBool` + `compare_exchange(AcqRel, Acquire)`, cleared by `StartedGuard` RAII (defused, never `mem::forget`). Shared cursor would double-emit. `start()` blocks until cancellation — MUST be spawned. `stop()` now fires `ctx.cancellation.cancel()` itself (no longer no-op), but does NOT await the spawned task — synchronous restart still requires `stop(); handle.await; start()`.
- **Poll loop shape.** `loop { cancel-check → poll → dispatch → sleep }` (flipped from `sleep → poll` in quick-hardening bundle). First poll runs immediately after `start()`. Pre-poll cancel check is a cheap `CancellationToken::is_cancelled()` atomic read — protects "start then immediately cancel" from running a stray poll. Post-poll sleep uses `tokio::select!` for cancel-safety.
- **PollConfig = constraint, not policy.** `PollConfig` declares the *integration-intrinsic timing constraints* (interval bounds, per-cycle timeout, failure policy). It does NOT decide scheduling policy: there is no `first_poll` knob, no `warmup_delay`, no fleet stagger. When-to-activate is runtime's job — runtime delays its call to `adapter.start()` for warmup/stagger. Do not re-add `FirstPoll`-shaped fields to `PollConfig`.
- **Jitter seed.** `trigger_seed(action_key, workflow_id, trigger_id)` FNV-1a hash — distinct per trigger instance. Prevents thundering herd when multiple workflow instances of the same action type start simultaneously. Seed is computed once at `start()` from `ctx.workflow_id` and `ctx.trigger_id`.
- **PollConfig validation.** `validate_and_clamp(logger, action_key)` runs once at `start()` before entering the loop. Clamps: `max_interval ≥ base_interval`, `backoff_factor ∈ [1.0, 60.0]` (also catches NaN/Inf), `poll_timeout > 0` (zero resets to 30s default). Misconfig gets a warn, not a fatal — credentials fail startup, wrong numbers do not.
- **Two trigger shapes.** `TriggerHandler::start` is either **setup-and-return** (webhook) or **run-until-cancelled** (poll). Sequentially awaiting multiple shape-2 triggers deadlocks. Always spawn.
- **Poll interval floor.** Clamped to `POLL_INTERVAL_FLOOR = 100 ms`. Tests must use ≥100 ms. Clamp logs one-shot warn via `ctx.logger` at `start()`. Per-cycle retryable/serialize/emit failures log via `ctx.logger` with per-kind `WarnThrottle` (30 s cooldown).
- **WebhookAction::State bounds.** `State: Clone + Send + Sync` only — no `Default`, no `Serialize`/`DeserializeOwned`. `on_activate` is the factory (required, no default). State persistence across restarts is a runtime concern (post-v1).
- **WebhookResponse.** `handle_request` returns `WebhookResponse` (not `TriggerEventOutcome`). `Accept(outcome)` = 200 OK. `Respond { http, outcome }` = custom HTTP response (Slack challenge, Stripe 200-within-5s). Response plumbed via `oneshot::Sender<WebhookHttpResponse>` attached to `WebhookRequest::with_response_channel`.
- **PollConfig.** `poll_config() -> PollConfig` replaces `poll_interval()` + `poll_timeout()`. Config struct: `base_interval`, `max_interval`, `backoff_factor`, `jitter`, `poll_timeout` (default 30s), `emit_failure` policy. Constructors: `PollConfig::fixed(interval)`, `PollConfig::with_backoff(base, max, factor)`. Adapter owns all timing: exponential backoff on idle, jitter for thundering herd prevention, floor clamping.
- **PollCursor<C>.** Wraps cursor with checkpoint capability. `Deref/DerefMut` to `C` for transparent access. `checkpoint()` saves progress during paginated polls. Non-paginated actions ignore checkpoints entirely.
- **PollOutcome enum.** `Idle` / `Ready { events }` / `Partial { events, error }`. No invalid combinations — each variant is one semantic. Fourth state (total failure) = `Err` from `poll()`. `PollResult` = `PollOutcome` + optional `override_next`.
- **Cursor rollback by outcome.** `Idle` → keep current. `Ready` ok → advance. `Ready` dispatch fail + RetryBatch → pre-poll. `Partial` → last checkpoint. `Err` → pre-poll.
- **DeduplicatingCursor<K, C>.** Wrapper cursor with bounded seen-key tracking (HashSet + VecDeque). Solves timestamp-granularity duplication (Gmail/Salesforce/Notion pattern). `filter_new(items, key_fn)` is the primary API. Cap default 1k, FIFO eviction.
- **PollAction::validate(ctx).** Called once before the poll loop. Default no-op. Return Err to abort activation.
- **PollAction::initial_cursor(ctx).** Called once before the first cycle. Default returns `Cursor::default()`. Override to start from "now" (prevent first-run flood).
- **RetryBatch + checkpoints.** RetryBatch ALWAYS rolls back to pre-poll, never to checkpoint. Paginated polls with RetryBatch re-fetch all pages on dispatch failure, including already-dispatched ones. At-least-once with maximum overlap. Use DeduplicatingCursor or downstream idempotency.
- **EmitFailurePolicy.** `DropAndContinue` (default), `RetryBatch` (restore cursor, re-fetch next cycle), `StopTrigger` (halt on first failure).
- **Cycle health is orthogonal to backoff.** `CycleOutcome.errored` drives `TriggerHealth::record_error`; `backoff` drives `consecutive_empty`. Dispatch-failure paths (RetryBatch / DropAndContinue total-loss / Partial dispatch fail / empty-events Partial with retryable) MUST set both — otherwise the dashboard misreads a failing trigger as `idle`. Silent regression before fix: RetryBatch hot-looped every `base_interval` with no backoff and reported idle.
- **`override_next` clamped by `max_interval`.** A rogue `Retry-After` of 1 h is capped at `max_interval` (never below `POLL_INTERVAL_FLOOR`). If you want an unconditional long sleep, raise `max_interval`.
- **`DeduplicatingCursor` batch sizing.** `max_seen ≥ 2 × max expected batch`. FIFO eviction happens inside `filter_new` as items are inserted — a batch larger than `max_seen` evicts items from the start of the same batch and leaks duplicates across the next cycle's boundary. `filter_new` debug-asserts on violation. `Deserialize` clamps `max_seen` to `≥ 1` (a persisted `0` would silently disable dedup).
- **Event limits.** `WebhookRequest::try_new` enforces `DEFAULT_MAX_BODY_BYTES` (1 MiB) + `MAX_HEADER_COUNT` (128). Returns `DataLimitExceeded` / `Validation`. Use `try_new_with_limits` for providers needing larger caps. Header keys lowercased at construction — `header()` is O(1). Duplicate keys collapse to last value.
- **Batch error semantics.** `BatchAction::process_item` → `Fatal` aborts the batch. Use `Retryable` for per-item failures the batch should capture and skip past.
- **DX macro requirement.** `impl_paginated_action!`/`impl_batch_action!` generate the `StatefulAction` impl — forgetting the macro → `register_stateful()` fails. One macro per type.
- **Resource downcast.** `ResourceActionAdapter::cleanup` downcasts `Box<dyn Any>` — mismatch is an engine bug, returns `Fatal`.
- **In-memory poll cursor.** `PollAction::Cursor` resets on every `start()`. Cross-restart persistence is a runtime concern (post-v1). `PollAction` trait docstring carries a `# Persistence — READ BEFORE SHIPPING` section warning against using current trait for payment/audit/high-value integrations without downstream idempotency — process restart either floods upstream (if `initial_cursor = Default`) or silently skips events (if `initial_cursor = now`).
- **PollConfig eviction candidates.** Follow-up plan `2026-04-13-poll-api-v2.md` may re-home the following to runtime policy: `jitter`, `backoff_factor`, `emit_failure` (per sdk-user/architect review — action author is not the source of truth for thundering-herd defense or dispatch semantics). Also deferred: `PollBudget` replacing advisory `max_pages_hint`, dropping `Default` bound on `Cursor`, cursor persistence (F8, blocked on runtime storage).
- **TriggerHealth.** Shared atomic state on `TriggerContext` as `Arc<TriggerHealth>` (not trait). Adapter writes via `record_success(n)` / `record_idle()` / `record_error()`. Runtime reads via `snapshot()` → `TriggerHealthSnapshot` (Serialize). All counters `Relaxed` ordering. No locks, no allocations per cycle.
- **`ctx.cancellation` / `ctx.emitter`** are `pub` fields on `TriggerContext` — known tech debt, tracked for an accessor refactor.
- **`ActionResult::Drop { reason }` vs `Skip { reason, output }`.** `Skip` marks the **entire downstream subgraph** as skipped — engine's `evaluate_edge` returns `false` for any edge, whole branch halts. `Drop` is narrower: this item produced no main output, but the broader branch continues processing other items in a stateful iteration, and parallel branches are untouched. Filter-style DX (future `FilterAction` in `ControlAction` family) must return `Drop`, never `Skip`, or the entire downstream subgraph disappears when one item fails the predicate. Added 2026-04-13 as Phase 0 correctness fix for the `ControlAction` work.
- **`ActionResult::Terminate { reason: TerminationReason }`.** Explicitly ends the whole execution regardless of parallel branches. `TerminationReason::Success { note }` → execution transitions to `Succeeded` with `ExecutionTerminationReason::ExplicitStop { by_node, note }` in audit log. `TerminationReason::Failure { code: Arc<str>, message }` → `Failed` + `ExplicitFail`. `code` is a placeholder `Arc<str>` until `ErrorCode` lands in Phase 10 of the action-v2 roadmap — callers treat it as opaque. Engine's `evaluate_edge` gates downstream edges on `Terminate` just like `Skip`; full parallel-branch cancellation with `ExecutionTerminationReason` recording is runtime/scheduler work tracked separately.
- **`ActionCategory` on `ActionMetadata`.** New field, `#[serde(default)]` for backward compat — legacy metadata without the field deserializes as `ActionCategory::Data`. Variants: `Data` / `Control` / `Trigger` / `Resource` / `Agent` / `Terminal`. Runtime dispatch does **not** depend on this — it is tooling metadata (UI grouping, validator rules, audit log filtering). The future `ControlActionAdapter` stamps `Control` / `Terminal` automatically so authors cannot forget.
- **`MultiOutput` join semantics pinned.** Docstring in `result.rs` now records: downstream nodes with multiple upstream edges fire when **all** emitted output ports carry data (same rule as `trigger_rule: all_success`). Absent ports mean "not emitted" and do not block unrelated downstream. `RouterAction` all-match mode relies on this rule.
- **`ControlAction` DX over `StatelessHandler` via adapter — not blanket impl.** Public, non-sealed trait in `control.rs` for flow-control nodes (If, Switch, Router, Filter, NoOp, Stop, Fail). Contract: `evaluate(ControlInput, &ActionContext) -> Result<ControlOutcome, ActionError>`. Desugar to `StatelessHandler` via explicit `ControlActionAdapter<A: ControlAction>` struct, following the `PollTriggerAdapter` / `WebhookTriggerAdapter` pattern. Coherence lives on the adapter type, not on `StatelessAction`, so future DX families targeting `StatelessHandler` can coexist via their own adapter types without a coherence budget. Concrete 7 nodes live **outside** `nebula-action` (downstream crate, TBD) — this crate owns only the trait contract, `ControlOutcome` / `ControlInput` / `TerminationReason` types, and the adapter. Community plugin crates may `impl ControlAction` directly.
- **`ControlActionAdapter::new` auto-stamps `ActionCategory`.** Clones the action's metadata, infers `ActionCategory::Terminal` for nodes with zero `outputs` (Stop, Fail) or `ActionCategory::Control` otherwise, caches in `Arc<ActionMetadata>`. Authors cannot forget to set the category — the adapter does it for them. The original metadata is preserved in all other fields; only `category` is rewritten.
- **`ControlInput` is owned, not borrowed.** Wrapper around `serde_json::Value` with typed getters (`get_bool`/`get_str`/`get_i64`/`get_f64`) + raw `get(pointer)`. Borrowed version (`&'a Value`) would conflict with the `+ 'static` bound on `StatelessHandler::execute`'s returned future — `async move` would copy anyway. `ControlInput` is `#[non_exhaustive]` for additive evolution.
- **`ControlOutcome` hides `Wait` / `Retry` / `Continue` from control authors.** Unified `#[non_exhaustive]` enum with exactly 5 variants (`Branch` / `Route` / `Pass` / `Drop` / `Terminate`). Prevents a control-flow node from accidentally returning a stateful-iteration variant that has no meaning in this context. Desugar is a `From<ControlOutcome> for ActionResult<Value>` impl — `Branch` → `ActionResult::Branch { alternatives: empty }`, `Route` → `MultiOutput { main_output: None }`, `Pass` → `Success`, `Drop` → `Drop`, `Terminate` → `Terminate`. Check `control.rs::From` impl before changing variants.
- **`ControlAction::evaluate` is native RPITIT, adapter is `#[async_trait]`.** Follow the same split as `StatelessAction::execute` (native `impl Future + Send`) vs `StatelessHandler::execute` (`#[async_trait]` for dyn compat). Do **not** put `#[async_trait]` on `ControlAction` itself — it would box the future on every call in the hot path. Erasure happens once at the adapter boundary.
