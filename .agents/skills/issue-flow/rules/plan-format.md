# Technical Plan Format

## Issue Comment Structure

When posting a plan to a GitHub Issue, use this exact structure. The HTML comment marker at the top enables idempotent updates.

```markdown
<!-- issue-flow-plan -->
## Technical Plan

> Auto-generated by `issue-flow` skill

### Summary

<2-3 sentence overview of the approach and key design decisions>

### Files

| File | Action | Description |
|------|--------|-------------|
| `path/to/file.ts` | Modify | <what changes and why> |
| `path/to/new-file.ts` | Create | <purpose of new file> |
| `path/to/test.test.ts` | Create | <what is being tested> |

### Implementation Steps

1. **<Step title>**: <Concrete description of what to do>
2. **<Step title>**: <Concrete description of what to do>
3. ...

### Test Strategy

- **Unit tests**: <what to unit test>
- **Integration tests**: <what to integration test>
- **Manual verification**: <optional manual checks>

### Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| <potential issue> | Low/Med/High | <how to mitigate> |
```

## Idempotent Update Rule

Before posting a new plan comment, check if an existing comment contains `<!-- issue-flow-plan -->`:

```bash
gh issue view <N> --json comments --jq '.comments[].body' | grep -l 'issue-flow-plan'
```

If found, **edit** the existing comment instead of creating a new one:

```bash
# Find the comment ID
COMMENT_ID=$(gh api repos/{owner}/{repo}/issues/<N>/comments \
  --jq '.[] | select(.body | contains("issue-flow-plan")) | .id')

# Update it
gh api repos/{owner}/{repo}/issues/comments/$COMMENT_ID \
  -X PATCH -f body="<new plan content>"
```

If not found, create a new comment with `gh issue comment`.

## Internal Plan Tracking

In addition to the Issue comment, maintain an internal task list via `TaskCreate` with:
- One task per implementation step
- Dependencies set via `addBlockedBy` where steps are sequential
- Clear descriptions that include file paths and specific changes
