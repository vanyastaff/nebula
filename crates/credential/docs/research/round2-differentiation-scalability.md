## ДИФФЕРЕНЦИАЦИЯ (Nebula vs Temporal)

**1. Кто владеет refresh/ротацией секрета — MOAT.** Temporal вообще НЕ делает credential-lifecycle: data-converter/codec шифрует payload client-side, ключи и refresh живут целиком в воркере пользователя, ядро Temporal про секрет ничего не знает (в repo `payloadSerializer` — это сериализация на уровне shard, без encryption-capability). Nebula делает ACTIVE refresh (rule 13-16, single-flight + CAS) первоклассным примитивом движка. Temporal жалеет, что так не сделал — у его пользователей «codec не дотянулся до воркера → токен не обновился» (ровно стена I в §22, Airflow client/server split). Это пустая ниша, верно выбранная.

**2. `policy(&State)->Decision` data-driven vs sealed enum — MOAT, при условии что F2 закрыт правильно.** Temporal внутри жёстко версионирует state-transitions через shard-lock + versioning (`renewRangeLocked`, CAS на RangeId) — у него НЕТ открытого мира протоколов, всё — фиксированные системные сущности. Nebula выбирает open-trait + одна обязательная `policy` (Finding 1, §19 F2). Это правильнее для плагинов. НО: пока `policy()` игнорирует `&State`, а резолвер роутит по hardcoded `C::KEY` (resolver.rs:209,536) — это MISTAKE-в-текущем-коде, не в дизайне. Temporal-урок здесь: единая точка решения, которую система ВСЕГДА вызывает (как shard-lock всегда держится), а не два рассинхронизирующихся зеркала (`Capabilities` bitflag + `policy.refresh`).

**3. Single durable timer / scheduler как порт — MISTAKE-риск.** Temporal — это, по сути, гигантский durable-timer engine: таймеры шардированы по `NumberOfShards`, масштабируются горизонтально per-shard. Nebula в 1.0 reactive-only и лишь «резервирует» `durable-timer` port (§19 q8, Finding 5). Это разумно для 1.0, НО лизинг-шедулер (`runtime/lease/scheduler.rs`) уже renew-ит на N% TTL — то есть proactive уже наполовину есть, в обход зарезервированного порта. Temporal это проходил: ad-hoc таймеры без durable-shard-инфраструктуры не масштабируются. Если порт не определить ДО появления второго прокативного пути — будет coordinator relocation обратно в движок.

**4. In-process registry vs внешний персистентный namespace-реестр — MOAT для cold-start, MISTAKE для multi-replica.** Temporal namespace-метаданные живут в persistence + кешируются, `ShardController` раздаёт владение шардами по нодам. Nebula держит типы credential в in-process registry (ADR-0091, no out-of-process/WASM). Для AI-discoverability и cold-start это лучше (нет сетевого round-trip на резолв типа). Но Temporal масштабирует именно тем, что namespace-плоскость — НЕ in-process: любой frontend-инстанс видит любой namespace. У Nebula P типов реестра живут в бинаре каждой реплики — см. стену 3 ниже.

**5. `OwnerScopedKey` (изоляция типом) vs namespace-as-runtime-entity — MOAT.** Temporal изолирует тенантов через namespace + per-namespace RPS/QPS (`NamespaceRPS`, `PersistenceNamespaceMaxQPS`, `EnableNamespaceFairness`) — это runtime rate-limiting, не типовая гарантия; confused-deputy ловится политикой, не компилятором. Nebula убирает `get(&str)` из store-порта (rule 9, Finding 3) — резолвер не может ВЫРАЗИТЬ нескоупленный load. Это строго сильнее: Temporal вынужден чинить cross-namespace leak проверками в рантайме, Nebula — по конструкции.

**6. Один CredentialRuntime pipeline vs разнесённые History/Frontend/Matching сервисы — MISTAKE-в-масштабе.** Temporal физически разделил плоскости (Frontend RPS-лимит ≠ History persistence-QPS ≠ Matching) ради независимого масштабирования и fairness. Nebula сводит acquire/refresh/revoke/test в один pipeline в одном крейте (D2). Для корректности и AI-discoverability — отлично. Для масштаба — единая точка, где горячий резолв и тяжёлый refresh делят один и тот же L1/CAS-путь без раздельного backpressure. Temporal именно это разнёс намеренно.

## МАСШТАБИРУЕМОСТЬ

**Как Temporal масштабирует (числа/пределы, deepwiki-grounded):** плоскость состояния шардируется по `NumberOfShards` (типично 512–4096, на крупных деплоях до десятков тысяч), `WorkflowID`→shard. Каждый shard — отдельный lock + `ioSemaphore` (контроль конкурентности LWT/CAS на персистентность). Лимиты — многоуровневые dynamic-config: глобальный `RPS`/`GlobalRPS`, per-namespace `NamespaceRPS`/`GlobalNamespaceRPS`/`MaxNamespaceRPSPerInstance`, persistence `PersistenceMaxQPS`/`PersistencePerShardNamespaceMaxQPS`, плюс `EnableNamespaceFairness` (fair-share demotion в low-priority bucket при перерасходе). History-cache (`HistoryHostLevelCacheMaxSize`, `HistoryCacheTTL`) гасит persistence-load на горячем пути. Контеншн на CAS истории решается shard-lock + versioning (`renewRangeLocked` дренит in-flight, обновляет `RangeId`); конфликты multi-cluster — `WorkflowResetter` форкает ветку. Секреты Temporal НЕ масштабирует на сервере вовсе — codec client-side, fan-out шифрования = воркеры пользователя, сервер видит opaque `DataBlob`. То есть его «credential-плоскость» масштабируется тем, что её на сервере нет.

**Первые 3 стены Nebula:**

**Стена 1 — L1 single-flight coalescer per `(owner, credential_id)` сериализует refresh при burst.** Механизм: одна durable-CAS-заявка на окно per ключ (стена B/§22). Ломается при сценарии «один шаренный credential, K параллельных воркфлоу, провайдер ротирует refresh_token каждый раз»: при burst Y одновременных исполнений все K сериализуются за одним coalescer-слотом; если refresh-POST к провайдеру = ~300-800ms, throughput по этому credential = ~1.5-3 refresh/s, остальные K-1 блокируются на single-flight wait. При тысяче воркфлоу на одном OAuth-credential это голова очереди. **Temporal так не упирается:** у него нет per-secret single-flight вообще — секрет резолвится client-side в каждом воркере независимо, контеншн размазан по `NumberOfShards`, а не по `(owner,cred)`.

**Стена 2 — `RefreshClaimRepo` CAS-contention при росте реплик.** Механизм: durable claim CAS, «только одна реплика рефрешит за окно» (D6). Ломается при Z репликах × высоком проценте expiring-credentials одновременно (синхронный expiry, напр. все токены провайдера живут ровно 1h → thundering herd на границе часа): Z реплик бьются за CAS на одни и те же claim-строки, CAS-retry растёт как O(Z) на горячих ключах, вторичный индекс credential→owner (adoption #9) для mass-revoke добавляет write-amplification. При Z≈десятки реплик и синхронном expiry окне — CAS-storm. `buffer_time`+`jitter` (adoption #7) это смягчает, но jitter в одном крейте без durable-timer-шардирования не равен шардированной таймер-плоскости Temporal. **Temporal так не упирается:** таймеры шардированы — каждый shard владеет своим подмножеством, нет общей горячей CAS-строки; владение раздаёт `ShardController`, CAS идёт per-shard `RangeId`, не per-global-claim.

**Стена 3 — in-process registry: P типов credential → cold-start и binary bloat на КАЖДОЙ реплике.** Механизм: ADR-0091, типы и `OAuth2ProviderConfig`-данные в бинаре in-process, no out-of-process. Ломается при росте P (число провайдеров/протоколов-семейств × per-provider configs): регистрация — eager на старте каждой реплики, cold-start растёт линейно по P, бинарь несёт все провайдеры даже если реплика обслуживает один тенант с одним провайдером; при N тенантах с непересекающимися наборами провайдеров всё равно O(P) в каждой реплике (нет partition по тенанту). Catalog-проекция схем (`schema_of` на каждый тип) — тоже O(P) на старте. **Temporal так не упирается:** namespace-метаданные в persistence, ленятся/кешируются с TTL, любой frontend видит любой namespace без запекания в бинарь — масштабирование числа namespace не трогает размер/cold-start бинаря. Митигация для Nebula (если не двигать ADR-0091): сделать registry lazy-per-type и partition-aware (резолвить provider-config из storage-данных, оставив в бинаре только код-per-protocol, а не config-per-provider) — это уже наполовину заложено формулой «code per protocol, config per provider» (§11), но config сейчас тоже registry-data in-process.

## ДИФФЕРЕНЦИАЦИЯ (Nebula vs n8n)

**1. Single-flight + durable RefreshClaimRepo CAS на refresh — MOAT.** DeepWiki подтвердил прямым ответом: в queue mode у n8n НЕТ распределённого лока. `OAuthService.refreshOAuth2CredentialById` делает fetch→`ClientOAuth2`.refresh→`update`, и «the last one to successfully update the database would prevail». Это ровно тот класс, который жжёт one-time refresh tokens (n8n #12742) и garbles данные. Nebula закрывает это `by construction` (L1 coalescer per `(owner,credential_id)` + durable CAS, §22-B). n8n именно тут жалеет — несколько воркеров одновременно ловят expired и гонятся за одним токеном.

**2. ArcSwap hot-swap / `CredentialRef` re-resolve vs «decrypt-on-demand» — частичный MOAT, частичная переоценка.** DeepWiki: n8n «does not explicitly cache decrypted credentials per-process» — каждый `authenticate`/`preAuthentication` дешифрует заново из БД. Это уродливо (CPU на каждый вызов: decrypt+Argon/AES), но как побочный эффект почти иммунно к stale-in-memory (n8n #1695 — баг был в долгой однопроходной операции, не в кэше). Nebula вводит SlotCell ArcSwap ради hot path — это быстрее, НО само создаёт риск stale-captured-guard, который n8n структурно не имеет. Здесь Nebula не «лучше», а делает другой trade-off и должна доказать инвалидацию тестом §22-C, иначе изобретает баг, которого у n8n нет.

**3. OwnerScopedKey (нет `get(&str)`) + связка через тип vs string-id + ролевые проверки — MOAT.** n8n резолвит по credential id и навешивает authz сверху (sharing/role); confused-deputy через resource-токен — реальная дыра, которую Windmill патчил (#9428). Nebula делает owner-isolation типом, не дисциплиной. Это сильнее именно потому, что у n8n authz «прибит сбоку» к read-share, отсюда G-баги (sharee может/не может reconnect).

**4. External secrets в hot path: env→provider→store + timeout+TTL+fail-closed — MOAT, но n8n тут даже впереди в одной детали.** DeepWiki: у n8n каждый внешний провайдер (Infisical) сам держит interceptor + свой token-refresh + кэш секретов — то есть провайдерский слой уже кэширует. Nebula формализует это контрактом (§10 rule 17, Airflow SecretCache урок) — это правильнее и единообразнее. MOAT в том, что у Nebula bounded-timeout+fail-closed — инвариант порта, а не свойство конкретного провайдера; у n8n это «повезло, что Infisical-клиент так написан».

**5. code-per-protocol-per-grant + config-per-provider vs `extends oAuth2Api` — паритет, не MOAT.** Это ровно n8n-модель (один `oAuth2Api` базовый + провайдеры как data/`extends`). DESIGN честно это признаёт (§19 #5). Не выдавать за преимущество: refresh-vs-reauth у n8n уже data-driven в `refreshOrFetchToken` (grantType + presence of refresh_token). `policy(&State)` — это догон до n8n, а не обгон. Реальный MOAT поверх — compile-gate (E0046), которого у n8n (TS) быть не может.

**6. Zero HTTP routes в credential crate + RefreshTransport как тип — MISTAKE-risk наоборот у n8n, MOAT у Nebula.** n8n смешивает `CredentialsHelper` (orchestration) и `OauthService` (HTTP-механика) в одном слое процесса; DESIGN §15 это уже скорректировал. Разделение seam'а правильнее n8n. Но осторожно: «zero routes» означает, что callback/kickoff живут в `nebula-api` — если воркер без api не может рефрешить (n8n queue-mode lesson, §22-I), это та же n8n-боль «keys live where refresh doesn't run». Здесь Nebula не наивна, но это стена (см. ниже).

## МАСШТАБИРУЕМОСТЬ

**Как n8n масштабирует (числа/пределы).** Queue mode: main + N воркеров через Redis (Bull), типично десятки воркеров; credentials — единицы сотен на инстанс у обычного self-host, тысячи у enterprise. Горячий путь резолва: НЕТ per-process кэша дешифрованных — каждый node-вызов = read из Postgres + decrypt. Refresh: НЕТ single-flight (DeepWiki) — координация только через атомарный `update` + `updatedAt`, «last writer wins». External secrets: провайдер сам кэширует (Infisical interceptor); enterprise external-secrets подтягиваются периодически, не на каждый резолв. То есть n8n масштабирует credential-плоскость в основном тем, что (а) credentials — холодные данные с редким write, (б) refresh редок относительно executions, (в) внешние провайдеры выносят TTL-кэш из ядра. Слабое место n8n признано самим кодом: refresh-storm и stale.

**Первые 3 стены дизайна Nebula:**

**Стена 1 — L1 single-flight coalescer сериализует burst на одном (owner,credential_id).** Механизм: per-key мьютекс/notify коалесцирует параллельные резолвы в один refresh. При burst Y одновременных executions, делящих ОДИН credential (типичный SaaS-аккаунт на весь tenant), все Y ждут одного HTTP refresh → задержка = refresh latency (часто 300мс–2с у медленных IdP), throughput по этому credential = 1/latency. При Y=500 шедулится разом (cron-кик) — 500 экзекуций стоят за одним POST. n8n этого не имеет, потому что не коалесцирует вообще (хуже на корректность, но НЕ создаёт точку сериализации — каждый воркер бежит независимо). Что делает n8n чтобы стены не было: ничего намеренного — просто independent refresh + «last wins»; платит корректностью. Nebula должна добавить `buffer_time`+`jitter`+proactive внутри reactive-кэша (§ Finding 5), иначе coalescer = bottleneck именно на популярных credential.

**Стена 2 — RefreshClaimRepo CAS contention при росте реплик + per-tenant fan-out O(N·M).** Механизм: durable CAS на claim — при Z репликах, конкурирующих за один claim-row, CAS-retry/abort растёт; при mass-revoke (`RevokeByToken`/`RevokePrefix`, §ad.9) проход по вторичному индексу credential→owner — O(M) на tenant, а массовая ротация по N тенантам = O(N·M) записей-тумбстоунов. При N=10k тенантов × M=200 credentials = 2M CAS-write на ротацию ключа — упрётся в write-IOPS Postgres. n8n не имеет этой стены, потому что не имеет durable claim вообще (нет координации) и не делает атомарную fan-out ротацию — ротация в n8n это ручной reconnect per-credential. То есть Nebula платит за корректность, которой у n8n нет, новой write-amplification стеной. Митигация: батчить tombstone-epoch на уровне owner (один epoch-bump инвалидирует все M), а не per-credential CAS.

**Стена 3 — in-process registry: P типов протоколов/провайдеров → cold-start + единый composition root для keys.** Механизм 3a: ADR-0091 in-process registry регистрирует каждый `Credential`/protocol/provider при старте; при P провайдеров (GitHub/Slack/… легко 100–300 в зрелом каталоге) — линейный cold-start init + рост бинаря (моно-крейт §21 F1 — весь subsystem компилируется/грузится вместе). Метрика: при P>~300 типов время старта воркера и size бинаря заметны, а compile-firewall потерян (touch contract → recompile ~6k LoC, Finding 5 D1). Механизм 3b (связан со Стеной из дифф-пункта 6): keys+RefreshTransport инжектятся ТОЛЬКО в `nebula-api` — engine/worker без api не рефрешат. При горизонтальном масштабировании воркеров без api-плоскости refresh централизуется в api → api становится единой точкой refresh-throughput (n8n #1732 / Airflow client-server split, §22-I). n8n обходит это иначе: каждый воркер сам полноценный (несёт OAuthService+ключи), refresh децентрализован — масштабируется добавлением воркеров, но ценой size copies ключей в каждом процессе. Nebula выбрала безопасность (ключи в одном корне) ценой централизации refresh — стена при росте числа воркеров, если api-refresh не шардирован.

## ДИФФЕРЕНЦИАЦИЯ (Nebula vs Apache Airflow)

**1. ExternalProvider chain: timeout+TTL-cache+fail-CLOSED (§10 rule 17) vs Airflow fail-OPEN.** — **MOAT, но с предупреждением.** Nebula правильно опознал боль (row H), но сделал противоположный fail-выбор. Airflow итерирует backend-chain и при сбое одного backend **логирует и идёт дальше** (fail-open): иначе временная недоступность Vault кладёт *парсинг всех DAG'ов*. У Airflow `SecretCache` помечен **experimental** и работает только во время DAG-parsing, а **не во время выполнения task'ов** — то есть на горячем пути исполнения кэша нет вообще, каждый task бьёт по backend. Nebula закладывает TTL-cache на резолве — это лучше. Но fail-closed на credential-резолве означает: упал Vault → встали все workflow, использующие external secrets. Airflow уже прошёл это и выбрал fail-open осознанно. Решение Nebula спорное: fail-closed корректнее с точки зрения безопасности (не подставить пустой secret), но в проде это превращает зависимость от внешнего SM в single-point-of-failure для всего движка. Airflow смягчает это chain'ом (env → backend), где env-fallback всегда работает.

**2. Worker/engine держит ноль ключей; refresh+Cipher инжектятся в единственном composition root nebula-api (§10 rule 8, row I).** — **MOAT.** Это ровно то, к чему Airflow пришёл болью: workers **не читают metastore напрямую**, а ходят через `ExecutionAPISecretsBackend` → Execution API server. Airflow добавил это поздно (AIP-44/AIP-72 task isolation), переписав путь резолва. Nebula закладывает это by-design с самого начала — engine/worker без reqwest и без ключей. Это выигрыш: Airflow годами имел worker'ы с прямым доступом к metastore (security/scaling-дыра), и context-detection (`SUPERVISOR_COMMS`, `_AIRFLOW_PROCESS_CONTEXT=server`) — это retrofit поверх старого дизайна. Nebula's single composition root чище.

**3. policy(&State)->Decision как единственный data-driven роутер refresh-vs-reacquire (§9, F2).** — **MOAT vs Airflow (но это копия n8n/Vault).** У Airflow **нет** активного OAuth-refresh как примитива connection-плоскости вообще — connection это статичный blob, refresh логики токенов нет в ядре; провайдеры (Google, AWS) делают refresh внутри hook'а/SDK ad-hoc. Nebula делает refresh first-class. Это реальный moat именно против Airflow (но не против n8n, у которого `refreshOrFetchToken` — та же форма). Airflow «пожалел» бы, что connection — мёртвый blob: пользователи Airflow городят custom secrets backend или внешний cron, чтобы ротировать токены.

**4. OwnerScopedKey: owner-isolation как ТИП, store без get(&str) (§10 rule 9, row F).** — **MOAT.** Airflow multi-tenancy исторически **слабая**: connections глобальны по conn_id в metastore, изоляция — на уровне RBAC/DAG-level permissions, не на уровне резолва secret'а. Confused-deputy (DAG автора X резолвит conn автора Y) в Airflow закрывается дисциплиной/policy, не типом. Nebula убирает саму возможность выразить unscoped load — это структурно сильнее. Airflow прошёл это болью (multi-team deployments → один namespace connections → утечки между командами; отсюда whole ecosystem обёрток типа per-team Vault mount paths).

**5. **Доп. узел (in-process registry, P типов):** cold-start реплики прогревает P credential-типов в registry; при P→большом binary-size и cold-start растут, а fail-closed cold-cache означает T-latency на первый резолв каждого типа после рестарта. Горизонталь: N API-реплик за LB, фоновые сервисы (scheduler/automations) отдельно, Postgres как единый источник состояния, Redis как шина событий. Воркеры stateless, поллят work pool, тянут BlockDocument по API на flow-run. Координация и backpressure — серверные concurrency limits на slot-based **timed leases** (слот освобождается по таймауту даже при падении процесса). То есть Prefect сознательно платит сетевым hop'ом за БД-как-единственную-истину и не пытается держать credential-состояние в процессе. Жёстких числовых пределов на Blocks нет — потолок = ёмкость Postgres/API; типичная стена — каждый flow-run = ≥1 синхронный read_block_document под нагрузкой fan-out.

**Первые 3 стены дизайна Nebula:**

**Стена 1 — `RefreshClaimRepo` CAS-contention при многих репликах.** Механизм: при burst истёкших токенов на популярный `(owner,credential_id)` все реплики гонятся за одной CAS-строкой; проигравшие ретраят/бэкоффят. При R реплик × частом OAuth-провайдере с коротким TTL (≈1ч) и синхронном пробуждении после рестарта — write-amplification и lock-contention на одной горячей строке. Ломается при десятках реплик на узком наборе горячих credentials. Prefect избегает этого тем, что не рефрешит вовсе (нет плоскости) — но честный аналог — это его серверные timed-lease слоты с разводкой через Postgres advisory-locks/Redis, а не per-row CAS-гонку. Nebula нужен provider-fingerprint-шардинг ключа клейма + jitter (уже в §17, но contention остаётся, если fingerprint грубый).

**Стена 2 — `policy(&State)` + резолв на горячем пути O(N·M) без кэша.** Механизм: per-tenant fan-out — N тенантов × M credentials, каждый dispatch зовёт resolve→project. Без TTL-кэша, партиционированного по `owner × scheme × provider-fingerprint` (он в §17 заявлен, но §22 признаёт «not yet met»), каждый шаг workflow = расшифровка State + project Scheme. При большом fan-out (тысячи параллельных нод на один credential) это CPU на decrypt + contention на single-flight. Single-flight per `(owner,cred)` сериализует именно успешный путь при burst Y одновременных потребителей одного токена — coalescer спасает refresh, но если кэш холодный, проектирование Scheme всё равно повторяется. Prefect снимает это серверными concurrency-limits + тем, что секрет резолвится один раз на flow-run, не на каждый шаг.

**Стена 3 — in-process registry: P типов протоколов/провайдеров → cold-start и binary, и нет cross-replica инвалидизации.** Механизм: при росте числа зарегистрированных типов/провайдеров (P) каждая реплика держит полный реестр в процессе; добавление/ротация провайдер-конфига требует согласованности между репликами, которой in-process не даёт — Prefect это решает тем, что реестр (Block types) живёт в Postgres и все реплики видят одну истину сразу. Метрика слома: при изменении provider-config в проде без рестарта реплики расходятся (одна знает новый scope/auth-url, другая нет) → недетерминированный refresh. Cold-start/binary — вторичная стена (монолитный крейт + все провайдеры линкуются статически). Prefect-эквивалент защиты: вынести provider-config-данные (не код) в durable store с pub/sub-инвалидизацией (его Redis-шина), оставив code-per-protocol в процессе.

Файлы прочитаны: `crates/credential/docs/DESIGN.md`, `crates/credential/docs/CONFERENCE.md`.

## ДИФФЕРЕНЦИАЦИЯ (Nebula vs Windmill) [grounded: deepwiki]

**1. Single-flight refresh coalescer + durable RefreshClaimRepo CAS — MOAT.** Windmill вообще не имеет single-flight: при доступе к ресурсу он inline-проверяет `expires_at` и дёргает `/oauth/refresh_token/{id}`, полагаясь только на DB-транзакцию для атомарности. DeepWiki прямо подтверждает: «does not explicitly detail a single-flight or locking mechanism to prevent concurrent refresh storms across multiple worker replicas». Это ровно n8n #12742 (concurrent refresh жжёт one-time refresh_token). Nebula's L1 coalescer per `(owner, credential_id)` + durable CAS — то, чего Windmill не сделал и о чём жалеет (Revolut #4582 — потеря rotated token).

**2. Atomic whole-State write-back rotated refresh_token (rule 13) — MOAT.** У Windmill refresh пишет «new refresh token (if available)» в `account`+`variable` таблицы раздельно, в одной транзакции, но без tombstone-epoch и без защиты от перезаписи устаревшим значением при гонке. Nebula делает CAS всей `State` целиком с tombstone — это закрывает #1-ю причину «reconnect loop». Windmill этим до сих пор болеет.

**3. OwnerScopedKey как ТИП (нет `get(&str)`) — MOAT.** Windmill обеспечивает изоляцию через `workspace_id` в каждом SQL — это discipline, не type-enforcement: одна забытая WHERE-клауза = cross-workspace leak. Их же confused-deputy/SSRF фикс (commit #9428, resolve token через permissioned path) — реактивная заплатка к ровно этому классу. Nebula делает изоляцию by-construction: резолвер не может *выразить* unscoped-load. Это по́зднее обучение Windmill, встроенное в тип.

**4. Reactive-only refresh при inline-on-access у Windmill — частично MISTAKE-риск.** Windmill refresh'ит строго при доступе (lazy, on hot path) — у Nebula та же reactive-модель 1.0. Но Windmill это делает БЕЗ кэша resolved-secret на hot path (DeepWiki: «no explicit caching layer... relies on direct database access»), что у них стена (см. ниже). Nebula добавляет `buffer_time`+`jitter`+`load_timeout` и ExternalProvider TTL-cache — это лучше, но reactive-only означает, что первый запрос после истечения платит латентностью refresh'а, как у Windmill. Здесь Nebula не радикально лучше — просто аккуратнее.

**5. In-process registry типов (ADR-0091) vs Windmill DB-as-registry — спорно, склоняется к MISTAKE на масштабе типов.** Windmill хранит resource_types/oauth-config как DATA в Postgres — добавление провайдера = строка в БД, ноль перекомпиляции, динамическая загрузка. Nebula: code-per-protocol компилируется в бинарь. Для protocol-families (~10) это MOAT (compile-time E0046 gate, типобезопасность). Но если число *провайдеров* поедет в код — это Windmill уже прошёл и ушёл в data; §11 «config-per-provider» это признаёт, но реализации пока ноль (CONFERENCE Finding 4: zero grep hits). Если провайдеры останутся data — MOAT; если просочатся в типы — повтор чужих граблей.

**6. RefreshTransport seam как type-invariant (не может нести OAuth2State/keys) — MOAT.** Windmill: воркеры сами декриптуют секреты workspace-ключом (`WORKSPACE_CRYPT_CACHE`), т.е. ключи живут в каждом воркере. Nebula держит Cipher+RefreshTransport в одном composition root (nebula-api), воркеры без ключей. Меньше поверхность атаки и нет «worker can't reach refresh path» (Windmill #1732). Windmill распределил ключи широко ради автономии воркеров — Nebula централизует ради безопасности.

## МАСШТАБИРУЕМОСТЬ [grounded: deepwiki + knowledge]

**Как Windmill масштабирует credential-плоскость.** Воркеры горизонтальны (`NUM_WORKERS`, обычно 1/контейнер), тянут джобы из Postgres через `FOR UPDATE SKIP LOCKED` (без lock-contention на очереди). Секреты резолвятся inline прямым SQL с `workspace_id`; воркер сам декриптует workspace-ключом, кэшируемым per-workspace в `WORKSPACE_CRYPT_CACHE` (инвалидация через LISTEN/NOTIFY `notify_workspace_key_change`). OAuth-refresh — lazy, при доступе к ресурсу, через API endpoint, внутри DB-транзакции; нет single-flight, нет фонового scheduler'а, нет кэша resolved-token на hot path. Практические пределы: refresh throughput ограничен Postgres write-транзакциями и числом одновременно истекающих токенов; координация мульти-реплик держится только на транзакционной атомарности (отсюда storm-баги); горячий путь резолва = 1+ SQL-round-trip на каждый доступ к секрету.

**Первые 3 стены дизайна Nebula:**

**Стена 1 — single-flight per `(owner, credential_id)` сериализует burst на ОДНУ credential.** Coalescer объединяет конкурентные refresh одной credential в один — отлично против storm. Но при «hot shared credential» (один сервис-аккаунт на 500 параллельных нод воркфлоу, токен истёк) все 500 встают за одним in-flight refresh + одним RefreshClaimRepo CAS-winner. Метрика слома: latency p99 первого запроса после expiry = время refresh (300ms–2s сетевого token-exchange) × все ждут; при QPS ≥ ~сотен на одну credential — очередь. *Windmill этого не имеет потому что* у него нет coalescer вовсе — он принимает storm (хуже на корректности), но не сериализует (нет single-point wait). Лечение для Nebula: proactive buffer (`buffer_time`+jitter, уже в §17) чтобы refresh случался ДО expiry вне горячего пути — но это 1.1, не 1.0.

**Стена 2 — RefreshClaimRepo CAS-contention при росте реплик × частоте refresh.** Durable CAS гарантирует «одна реплика refresh'ит за окно», но сам CAS — это write к одной строке-claim per credential. При N реплик, каждая видит expiry одновременно (нет jitter на сам claim-attempt) → N CAS-попыток на одну строку, N-1 откатываются. Механизм слома: на claim-репозитории поверх Postgres это write-contention/aborted-txn rate растёт с N реплик; при массовой rotation (tenant с M credentials истекают в одно окно) → O(M) горячих строк × N реплик = M·N CAS-attempts всплеском. *Windmill избегает* тем, что не координирует вовсе (каждый воркер refresh'ит что нашёл) — ценой storm-багов, но без CAS-узла. Лечение: jitter на claim-attempt + secondary-index credential→owner (§ CONFERENCE adoption 9) для batched-claim вместо per-credential.

**Стена 3 — reactive-only резолв без proactive scheduler → латентность на hot path при per-tenant fan-out.** §22 row H/rule 17 даёт ExternalProvider TTL-cache, но reactive-only (ADR-0084) означает: refresh триггерится политикой при резолве. При большом fan-out (тенант запускает воркфлоу с K нод, каждая резолвит свою credential, часть истекла) первый резолв каждой платит refresh-латентностью синхронно на горячем пути исполнения — ровно то, чем страдает Windmill (inline-on-access, no hot-path cache). Метрика: при K истёкших credential в одном воркфлоу-тике cold-start воркфлоу растёт линейно K × refresh-RTT. *Windmill «решает» это никак* (та же боль — DeepWiki подтверждает отсутствие кэша); Airflow решает `SecretCache`+context-aware chains. Nebula's TTL-cache помогает повторам, но первый промах каждой credential — стена до 1.1 proactive scheduler (durable-timer port уже зарезервирован — правильный задел).

Бонус-стена (in-process registry, ADR-0091): P типов протоколов компилируются в бинарь — cold-start/binary-size растёт с P; но т.к. families ~10 и провайдеры — data, это упрётся только если провайдеры просочатся в код (см. Дифференциация п.5). Windmill держит это data-в-Postgres и не имеет такой стены вовсе.

## ДИФФЕРЕНЦИАЦИЯ (Nebula vs Kestra)

**1. Активный lifecycle (refresh/rotation/lease) vs статичные секреты — MOAT, крупнейший.**
Kestra трактует секреты как статичные значения. `SecretService` декодирует `SECRET_*` env vars в `Map<String,String>` один раз на `postConstruct`; в коде нет refresh, нет OAuth-rotation, нет lease TTL — провайдер сам управляет жизненным циклом, Kestra только читает. Nebula делает кредл активным примитивом движка: `policy(&State)->Decision{Use|Refresh|ReAcquire|Lease|Revoke}`, `RefreshClaimRepo` CAS, `LeaseHandle`, атомарная перезапись rotated `refresh_token` (§10 rule 13). Это ровно та ниша, где Kestra-юзеры идут на внешний Vault и всё равно ловят expiry. Nebula здесь объективно глубже. Caveat: Kestra сознательно делегировал refresh внешнему SM — это «не баг, а scope», и Nebula берёт на себя сложность, которую Kestra умышленно вынес наружу.

**2. OAuth Plane Law / token-exchange внутри движка — MOAT с оговоркой.**
У Kestra нет встроенного OAuth-token-exchange как кредл-механики вообще: OAuth-провайдеры — это плагины задач или внешний SM. Nebula строит Plane A/B разделение, PKCE-kernel, `RefreshTransport` seam. Это даёт UX «подключи Slack кнопкой» из коробки, чего Kestra OSS не имеет. Caveat-MISTAKE-риск: Kestra осознанно НЕ держит HTTP/SSRF-поверхность в ядре — а Nebula втягивает connect-layer SSRF resolver, dual-kickoff дедупликацию и т.д. внутрь. Поверхность атаки и поддержки, которой у Kestra просто нет.

**3. Schema-from-types + values-only vs free-form KV/JSON — MOAT.**
Kestra KV store и `secret()` отдают строки/JSON-blobs без типовой схемы (`kv('key')`, subkeys парсятся как JSON ad-hoc). Nebula выводит схему из Rust-типов (`Properties: HasSchema`), хранит только values, валидирует при create. Это даёт типобезопасные формы и compile-time гарантии, которых у Pebble-`secret()` нет. Kestra тут реально слабее на DX/корректности.

**4. Namespace-наследование секретов — ОТСУТСТВУЕТ у Nebula, MISTAKE-риск.**
Kestra даёт иерархию: секрет в parent namespace наследуется всеми child namespaces — это killer-feature для команд (один раз настроил на верхнем уровне). Дизайн Nebula строит жёсткую `OwnerScopedKey` изоляцию by-type без `get(&str)` — отлично для confused-deputy, но в нём НЕТ модели иерархического/наследуемого scope. Для multi-team self-hosted это будет ощущаться как регресс vs Kestra; «owner» плоский. Это не security-дыра, но product-gap, который Kestra уже закрыл.

**5. Single public crate + sealed-ish protocol families vs open plugin model — спорно, MISTAKE-риск.**
Kestra-плагины (включая SM-интеграции: Vault, AWS SM, Azure KV, GCP, Elasticsearch, read-only backends) — открытая внешняя экосистема, добавляется без трогания ядра. Nebula 1.0: in-process registry, ~10 protocol families, F3 (open-world escape hatch) ещё не решён. Если набор семейств окажется sealed, то «новый протокол кредла» = PR в ядро, тогда как у Kestra это сторонний плагин. Kestra намеренно сделал SM-список расширяемым данными/плагинами — Nebula рискует упереться в это при росте числа провайдеров.

**6. External secret manager как first-class backend — Kestra впереди (MISTAKE-риск для 1.0).**
Kestra EE из коробки интегрирует Vault/AWS SM/Azure/GCP с «optional caching reduces API calls». У Nebula это `ExternalProvider` chain (env→provider→store, timeout+TTL+fail-closed) — концептуально та же идея и архитектурно даже строже (fail-closed, bounded timeout — Airflow-урок, который Kestra документирует слабее). Но в Nebula это пока spec, у Kestra — shipped EE-фича. На паритет надо дойти.

## МАСШТАБИРУЕМОСТЬ

**Как Kestra масштабирует credential-плоскость (grounded: deepwiki).**
Ключевое: горячий путь резолва секрета у Kestra — это in-memory lookup, не сетевой вызов. `SecretService` — singleton на процесс; все `SECRET_*` декодированы в `Map<String,String>` на старте; `secret()` в Pebble бьёт прямо в эту map → O(1), без I/O, без локов. При тысячах executions узкое место смещается на CPU рендеринга выражений и сетевой I/O задач, НЕ на резолв секрета. Координации между репликами по секретам нет вообще — каждая реплика держит свою копию map (env-driven, поэтому консистентна by construction). Для EE-внешних SM есть optional caching (TTL) поверх Vault/AWS SM. KVStoreService — namespace/tenant-scoped, детали локинга не раскрыты, но KV — не на refresh-CAS-пути. Чисел refresh-throughput у Kestra нет, потому что **refresh-плоскости нет**: нет single-flight, нет CAS-contention, нет per-tenant fan-out — это цена статичной модели. Multi-tenant EE: dedicated backends per namespace/tenant/instance, изоляция через scoping, не через runtime-координацию.

**Первые 3 стены дизайна Nebula при росте.**

**Стена 1 — горячий путь резолва: per-resolve work vs Kestra O(1) map-hit.**
Kestra: секрет = указатель в process-local map, ~0 на резолв. Nebula: каждый slot-резолв проходит `OwnerScopedKey` store-query + `policy(&State)` + project()→Scheme. При burst в Y tasks/sec, где каждый task имеет K кредл-слотов, это Y·K резолвов с (как минимум) decrypt + policy-eval, а при cache-miss — store round-trip. Метрика слома: при Y·K, превышающем пропускную способность store/decrypt (напр. >5–10k резолвов/сек на реплику с per-resolve decrypt), резолв из «бесплатного» (как у Kestra) становится bottleneck'ом DAG-парсинга. **Что делает Kestra:** статичная in-memory map + log-masking, ноль per-resolve crypto на горячем пути. **Митигация Nebula (уже в дизайне, но надо мерить):** AWS-style cache partition `owner × scheme × provider-fingerprint` + `SlotCell` ArcSwap, чтобы повторный резолв был map-hit, а не store-hit — но это надо сделать дефолтом, иначе стена реальна.

**Стена 2 — single-flight coalescer per `(owner,credential_id)` + RefreshClaimRepo CAS-contention при burst и при росте реплик.**
Это плоскость, которой у Kestra нет, поэтому она целиком на Nebula. Механизм: при истечении общего кредла, на который висит burst из N конкурентных executions одного тенанта, single-flight сериализует их все на один refresh (правильно против n8n #12742 burn-token), НО под нагрузкой это значит, что N задач блокируются на хвосте одного RefreshTransport round-trip (сетевой OAuth POST, сотни мс). Метрика слома: при N конкурентных потребителей × латентность refresh L, эффективная задержка = L на всю группу; при popular shared credential и L~300ms+ это видимый stall. Параллельно при Z репликах durable `RefreshClaimRepo` CAS — это write-contention на одну строку (owner,cred): при Z растущем и коротком refresh-окне CAS-loop retry/abort деградирует (классический hot-row). **Что делает Kestra:** проблемы нет — нет refresh, нет shared-mutable-credential CAS. **Дизайн Nebula:** durable-timer port + buffer_time+jitter (proactive внутри reactive) сдвигает refresh с пикового момента — это и есть нужное лекарство, но оно помечено как 1.1/частично; до него стена остаётся.

**Стена 3 — per-tenant fan-out при rotation/mass-revoke и in-process registry cold-start.**
(a) Fan-out: при rotation одного кредла надо обновить все `SlotCell` epoch'и у активных resource-pools, плюс `RefreshClaimRepo` secondary index credential→owner для mass-revoke. При N тенантов × M кредлов × P resource-bindings, mass-revoke/rotation — это O(N·M·P) обновлений; при tenant-wide revoke (compromise) фан-аут на все live guards упирается в проход по всем активным слотам. Метрика: tenant с большим M и широким биндингом → revoke latency растёт линейно по числу активных guard'ов. **Kestra:** revoke = убрать env/SM-запись, реплики просто перестанут находить на следующем рестарте/чтении — нет live-guard графа для обхода (цена — нет мгновенного активного отзыва, но и нет фан-аут стены). (b) In-process registry: P типов протоколов/провайдеров компилируются в бинарь (ADR-0091, no WASM/out-of-process). При росте P растёт cold-start регистрации и размер бинаря; что важнее — новый протокол требует пересборки ядра, тогда как Kestra грузит SM-backend как плагин/данные. Метрика: P → линейный binary/cold-start, и ноль runtime-расширяемости. **Что делает Kestra:** провайдеры/SM-backends — данные и плагины, добавляются без пересборки ядра; список SM открыт by design.

## ДИФФЕРЕНЦИАЦИЯ (Nebula vs Restate)

**2. Единый pipeline с policy(&State) vs durable-функция как код.**
Restate сериализует решения в журнал: «вызвать token endpoint» — это journaled side-effect (`ctx.run(...)`), идемпотентность через replay. Refresh-решение у Restate **не централизовано** — оно размазано по коду каждой функции. Nebula централизует через `policy(&State)->Decision`. Restate тут прошёл дальше: durable execution **поглощает** проблему single-flight/exactly-once, которую Nebula решает точечно.

**3. In-process registry vs распределённый partition-log.**

Restate решил multi-replica координацию в фундаменте; Nebula её отложила.

**4. Owner-isolation как тип vs partition-as-isolation.**
У Restate изоляция/co-location идёт через `PartitionId` — ключ (например, virtual-object key) детерминированно мапится в партицию, и весь стейт этого ключа живёт в одном `PartitionStore`. Подход Nebula — **MOAT по безопасности**: confused-deputy закрыт типом, а не дисциплиной (Restate не даёт этой гарантии — изоляция тенантов в Restate это application-level concern, partition не равно tenant). Nebula's owner-scoping не co-locates по умолчанию — это решение про безопасность, не про locality.

**5. RefreshTransport seam vs journaled side-effect.**
Nebula выносит HTTP за узкий typed seam (`Url+form → capped bytes`, connect-layer SSRF resolver, MUST). Restate любой внешний вызов оборачивает в `ctx.run()` и журналирует результат — SSRF/egress-контроль **не встроен**, это снова код функции. Restate бы выиграл от этого seam, но его философия «durable-функция = произвольный код» делает egress-allowlist неестественным.

## МАСШТАБИРУЕМОСТЬ
[grounded: deepwiki для архитектуры Restate; числовые пределы Nebula = knowledge-only из дизайна]

**Как Restate масштабирует (из deepwiki + архитектурное знание).**
Restate **не масштабирует credential-плоскость — у него её нет в ядре**; он масштабирует durable execution, и любая identity/secret живёт снаружи (внешний SM или env). Масштабируется так: horizontal partitioning через Bifrost (число партиций фиксируется, реплики добавляются), leadership per-partition (один лидер пишет, без global lock), `DurabilityMode` (Balanced/SnapshotOnly/ReplicaSetOnly) регулирует когда триммить лог. Горячий путь — локальный `PartitionStore` (RocksDB-class), не сетевой. Refresh-throughput в модели Restate = просто журналируемые side-effects, размазанные по партициям → масштабируется линейно с числом партиций, exactly-once гарантирован replay'ем, без отдельного single-flight-механизма. Multi-replica координация — встроена (replica-set membership), per-tenant fan-out отсутствует как понятие (tenant ≈ key ≈ partition).

**Три стены, в которые упрётся Nebula:**

**Стена 1 — `RefreshClaimRepo` CAS contention при multi-replica.** Механизм: reactive refresh = L1 single-flight per `(owner, credential_id)` **в процессе** + durable CAS на shared store между репликами. На одной ноде L1 coalescer спасает. Как только Z реплик читают один store, CAS на горячем credential становится точкой сериализации: при burst (N тенантов разом проснулись на расписании) → Z реплик бьются за один CAS-слот → spin/retry, p99 refresh-latency растёт линейно с Z. Ломается при ~Z≥3-4 реплик + коррелированный burst (cron-триггеры в :00). **Restate не имеет этой стены:** один лидер per partition пишет без contention — CAS заменён на leadership, конкуренция исключена by design, а не разрешена retry-loop'ом.

**Стена 2 — O(N·M) на mass-revoke / rotation fan-out.** Механизм: при ротации провайдера или mass-revoke надо обойти N тенантов × M credentials. CONFERENCE adoption #9 уже добавил secondary index credential→owner — но fan-out на `SlotCell` epoch-bump всё равно O(затронутых слотов), а in-process registry держит это в памяти одной ноды. При N·M порядка 10^5-10^6 активных bindings mass-revoke становится stop-the-world проходом по памяти + store-записям без партиционирования. Метрика излома: revoke-latency растёт линейно с числом активных слотов; при ~10^5 bindings один mass-revoke = секунды блокировки горячего пути. **Restate не имеет этой стены:** fan-out естественно партиционируется — каждый затронутый key в своей партиции, revoke применяется как обычная journaled-команда параллельно по партициям, нет единой in-memory структуры для обхода.

**Стена 3 — in-process registry: cold-start и binary при росте числа типов/тенантов.** Механизм: ADR-0091 держит все P типов протоколов/провайдеров + регистрацию в одном процессе. Два под-предела: (a) **типы** — code-per-protocol-per-grant + config-per-provider означает, что рост числа провайдеров — это data (ок), но рост числа *семейств протоколов* — это код в binary; P семейств → линейный рост binary + время регистрации на cold-start, и single-node = нет горизонтального шардинга самого registry. (b) **горячий резолв** — `ExternalProvider` chain (env→provider→store) с per-call timeout + TTL-cache fail-closed: при cache-miss-storm (TTL истёк у K тенантов одновременно) chain делает K сетевых вызовов к внешнему SM с одной ноды → upstream rate-limit / timeout-каскад. Ломается при K одновременных miss > rate-limit внешнего SM (Airflow ровно это — стена H в §22). **Restate не имеет этой стены:** нет centralized registry на горячем пути — стейт партиционирован, cold-start ноды поднимает только её партиции (не весь мир), а identity-резолв вынесен наружу и не является single-node bottleneck'ом ядра.

**Сводно:** все три стены Nebula — следствие одного выбора: Nebula тащит credential-координацию **внутрь** single-node-ориентированного движка (это и есть её MOAT по DX/безопасности), тогда как Restate **вытолкнул** identity наружу и отдал координацию партиционированному логу. Nebula платит за moat тремя механизмами (CAS, in-memory fan-out, in-process registry), которые у Restate растворены в фундаменте. Для 1.0 single-node/self-hosted это правильный размен; первая же multi-replica инсталляция упрётся в Стену 1.

## МАСШТАБИРУЕМОСТЬ

**Как Vault масштабирует credential-плоскость (числа/механизмы):**
- Лизы: два `BarrierView` (`idView`/`tokenView`) + in-memory `sync.Map` (`pending`/`nonexpiring`); `leaseCount` с warning-порогом `maxLeaseThreshold = 256k`. Прод-инсталляции уходят за миллионы лизов, но это и есть зона боли — каждый лиз = таймер + storage-entry.
- Revoke: `fairshare.JobManager`, ~200 воркеров по умолчанию, exponential backoff (`maxRevokeAttempts = 6`), затем `irrevocable`-маркировка — лиз перестаёт жечь ресурсы.
- Revoke-storm: `RevokePrefix` — ленивая массовая ревокация (mark-for-expiry, не sync), чтобы не положить кластер.
- Multi-replica: lease-state в barrier-storage, реплицируется через storage; standby при старте входит в restore-mode и rebuild'ит in-memory лизы worker-pool'ом; `invalidate` синхронизирует in-memory при изменении storage. Перфоманс-standby обслуживают read-путь, write/renew проксируются на active.

**Первые 3 стены дизайна Nebula:**

**Стена 1 — O(N·M) cold-start in-process registry + per-(owner,cred) state fan-out.** Registry держит P типов протоколов (мало, ~10), но ХОТ-резолв материализует `SlotCell`/`CredentialRef` per (owner × credential_id). При N тенантов × M активных credential'ов живущие в памяти ArcSwap-ячейки растут как O(N·M); при N·M ~ сотни тысяч это RSS-стена и GC-подобное давление на Arc-rc, плюс cold-start engine'а, который должен warm'ить bind-population. *Vault не имеет:* у него лизы в barrier-storage + restore-mode worker-pool, in-memory только pending-таймеры, а не полная material каждого секрета.

**Стена 2 — single-flight per (owner,credential_id) сериализует refresh при burst.** При шумном tenant'е, где одна OAuth-credential шарится M параллельными workflow-нодами и истекает разом, coalescer корректно схлопывает в один refresh — но все M исполнений *блокируются* на этом single-flight до CAS-persist. Метрика слома: refresh-latency провайдера T_r × число одновременно-истёкших горячих credential'ов K → хвост латентности исполнений = K независимых T_r-столлов, при T_r ~1-2с и burst K в сотни — видимый stall фронта. *Vault не имеет:* fairshare раздаёт revocation/renew по воркерам (шард, не очередь-за-строкой); один медленный backend не стопорит остальные.

**Стена 3 — RefreshClaimRepo CAS-contention при Z replica + thundering revoke.** Durable CAS-claim хорош для одного-победителя-per-window, но при Z replica, проснувшихся на одном expiry-окне горячей credential, все Z бьются за одну строку → CAS-retry-storm, write-amplification на storage. И при mass-revoke (`RevokeByToken`/`RevokePrefix` из §21 п.9) вторичный индекс credential→owner — это O(matched) последовательных tombstone-CAS, без fair-share воркеров; при revoke десятков тысяч — лавина CAS на одну партицию. Метрика слома: Z replica × hot-window collisions → CAS-abort rate, деградирующая суперлинейно. *Vault не имеет:* centralized `ExpirationManager` на active-узле — единственный писатель лизов, нет multi-writer CAS-гонки; storm гасится `fairshare` + `irrevocable`, а не retry-loop'ом.

Сводно: Nebula выиграла горячий путь (in-process, нулевая латентность резолва) ценой того, что Vault решил централизацией — координация записи под нагрузкой. Рекомендация на 1.0: зарезервировать `durable-timer` порт (уже в §21 п.7) и спроектировать RefreshClaimRepo с шардингом claim'ов по `owner × scheme`, а не одной строкой per credential, чтобы Z-replica contention деградировала линейно, а не суперлинейно — это Vault'овский fairshare-урок, перенесённый в CAS-модель.

## ДИФФЕРЕНЦИАЦИЯ (Nebula vs AWS SDK for Rust)

Каждая реплика/клиент рефрешит независимо — для STS это норм (refresh идемпотентен, новый токен не отзывает старый). Но при провайдерах, которые **ротируют** refresh_token (Nebula §22 row A: n8n #30345, Revolut #4582), N реплик без координации = один сжигает токен другого. Идея правильнее (lease без expires_at, ReAcquire vs Refresh — Finding 1/2), но если `policy()` не тривиальна, это дороже AWS-time-check. MOAT по корректности, потенциальный MISTAKE по стоимости горячего пути, если policy разрастётся.

**3. Cache partition key — MISTAKE-риск, AWS уже наступил.** AWS partition = HMAC от `(access_key_id + secret + bucket)`; общий `IdentityCachePartition` для дефолтной цепочки. Nebula §21.7 предлагает `owner × scheme × provider-fingerprint`. Но §10 rule 9 `OwnerScopedKey` = length-prefixed `owner_id + id` — это **другая** гранулярность, чем cache-partition. AWS показал, что несовпадение partition-ключа и identity-ключа = либо ложные cache-промахи (over-partition), либо межклиентский leak (under-partition); S3Express специально включил bucket в ключ, чтобы не отдавать чужой scoped credential. Nebula должна свести OwnerScopedKey и cache-partition к **одному** деривату, иначе два источника правды (та же desync-болезнь, что §9 Capabilities-bitflag vs policy.refresh).

**4. Один Mutex вокруг кэша — MISTAKE, который Nebula может унаследовать.** DeepWiki прямо: `S3ExpressIdentityCache` = «single mutex protecting the entire cache, not a per-partition lock». Это узкое место AWS под burst многих **разных** identity (все блокируются на одном lock даже при cache-hit). Nebula §9 single-flight per `(owner,credential_id)` — правильно гранулярно по записи, но реестр coalescer'ов (DashMap/Mutex<HashMap>) сам станет тем же глобальным lock'ом, если не шардирован. Здесь Nebula **может** оказаться лучше AWS — но только если явно избежит single-Mutex; дизайн это пока не специфицирует.

**5. `RefreshTransport` как тип-инвариант (не может нести `OAuth2State`/ключи) — MOAT.** AWS `IdentityResolver` — общий trait, провайдер сам делает сеть; SSRF-защита оставлена приложению, DNS-rebind не предотвращён на уровне типа. Nebula §10 rule 8 (connect-layer resolver = MUST, узкий seam) структурно сильнее. AWS обжёгся на IMDS SSRF (отсюда IMDSv2 с hop-limit/токеном) — Nebula закрывает это by-construction, чего у AWS-типов нет.

**6. ExternalProvider chain с per-call timeout + TTL-cache + fail-closed — паритет, не MOAT.** Это ровно AWS `CredentialsProviderChain` + `load_timeout` + кэш. Nebula §10 rule 17 переоткрывает то, что AWS уже стандартизировал. Не преимущество — это «догнали правильную базу». Риск MISTAKE только если fail-closed для **env**-провайдера сделает cold-start хрупким (AWS специально fail-open по цепочке до последнего звена).

## МАСШТАБИРУЕМОСТЬ

**AWS SDK for Rust в проде.** Credential-плоскость почти бесплатна на горячем пути by design: после первого резолва identity лежит в `LazyCache`/`IdentityCache`, и каждый запрос — это lock + hashmap-lookup + time-check (`now + buffer_time vs expires_at`), сетевой вызов провайдера только при miss/expiry. Refresh throughput управляется `buffer_time` (проактивный refresh до истечения, снимает latency-spike), `load_timeout` (рефреш не блокирует бесконечно), single-flight через `ExpiringCache::get_or_load` (один loader на identity). Координация: **никакой** межпроцессной — кэш per-client/per-process, каждая реплика рефрешит сама. Числа: десятки-сотни STS-сессий на процесс, refresh раз в ~15-60 мин на identity, SigV4 на горячем пути не трогает провайдер вообще. Масштаб «много identity» решается partition-ключом, а не fan-out'ом — нет per-tenant координации, потому что у AWS нет понятия «тенант рефрешит чужой токен».

**Три стены, в которые упрётся Nebula:**

**Стена 1 — реестр single-flight coalescer'ов сериализует burst.** Механизм: `L1 single-flight per (owner,credential_id)` живёт в карте; если карта за одним `Mutex`/`RwLock`, то при burst Y одновременных резолвов **разных** credential все блокируются на вставке/lookup ключа — ровно AWS-болезнь «single Mutex protecting the entire cache». Метрика слома: при ~10³–10⁴ конкурентных резолвов/сек разных `(owner,cred)` p99 резолва растёт из-за lock-convoy, хотя сами записи независимы. AWS живёт с этим, потому что у него identity мало; у Nebula `N тенантов × M credentials` ключей. **Что делает AWS:** ничего лучше single-Mutex — значит Nebula обязана **обогнать** AWS: шардированный `DashMap`/per-shard lock, а не один. Дизайн это не фиксирует — добавить в §9 как инвариант.

**Стена 2 — `RefreshClaimRepo` CAS contention при росте реплик и mass-refresh.** Механизм: durable CAS — это запись в storage (Postgres-row/conditional-put) на каждый refresh-claim. При Z репликах и burst истечений (напр. провайдер с коротким TTL, 1000 credentials истекают в одном окне) все Z реплик бьются за CAS по тем же строкам → write-amplification + retry-storm на БД. Метрика слома: refresh-rate × Z реплик × contention → CAS p99 и DB write IOPS превышают бюджет уже при сотнях credential/мин и >3-4 репликах без jitter. §21.7 `jitter` помогает, но claim — это durable round-trip, которого у AWS нет вовсе. **Что делает AWS:** убирает проблему целиком — нет durable claim, каждая реплика рефрешит in-memory локально (для STS это безопасно). Nebula платит durable CAS только потому, что нужна ротация-safe-семантика; цена — масштаб ограничен пропускной БД. Митигация: `buffer_time + jitter` (§21.7) + secondary index credential→owner (§21.9), но фундамент остаётся БД-bound.

**Стена 3 — `policy(&State)` + проекция `refresh_error`/`last_refresh_at` на горячем пути резолва.** Механизм: Nebula зовёт `policy(&State)` на **каждом** резолве (§9, для всех strategy, не только inline-expiry), плюс §21.8 пишет observable-поля в State. У AWS горячий путь — арифметика под мьютексом без вызова policy и без записи в state при cache-hit. Метрика слома: при per-tenant fan-out (один workflow трогает K credential на шаг, R шагов/сек) это `O(K·R)` вызовов policy + потенциальных state-mutation; если `policy` дешёвая — ок, но как только она читает lease/TTL/provider-config-data, hot-path-стоимость на порядок выше AWS time-check. Дополнительно `in-process registry: P типов протоколов → cold-start/binary` (ADR-0091): рост числа зарегистрированных Credential-типов раздувает реестр и cold-start, тогда как AWS — фиксированный набор провайдеров, скомпилированных статически. **Что делает AWS:** держит горячий путь = чистый time-check, policy-эквивалент (`buffer_time`) — константа в конфиге, не вызов на запрос; и не пишет в state при hit. Nebula должна гарантировать, что `policy` — pure/cached и cache-hit **не** трогает durable state, иначе каждый резолв = запись.

АТАКИ-НА-МАСШТАБ / ПОЗИЦИОНИРОВАНИЕ

1. L1 single-flight НЕ устраняет хвост — он сериализует burst за L2 TTL. Механизм: при N конкурентных нод (репликах) на один credential ровно одна нода — Winner на L1; все остальные на каждой реплике коалесируются локально и возвращают `CoalescedByOtherReplica` → перечитывают State. Но cross-replica путь (`try_acquire_l2_with_backoff`, coordinator.rs:705) — это polling-CAS с `MAX_ATTEMPTS=5`, сном `min(existing_expires_at, now+5s)+jitter(0..100ms)`. При K реплик-контендеров: каждая проигравшая спит до ~5s, перечитывает state, и если Winner ещё не дописал (его `refresh_timeout=8s` > backoff cap 5s) — снова try_claim. Слом: если реальный refresh IdP занимает >5s (медленный провайдер, p99), backoff-сон истекает ДО релиза claim, контендер делает повторный try_claim против ещё-живого heartbeat-claim, и за 5 попыток (потолок ~25s wall-clock) упирается в `ContentionExhausted`. Метрика слома: refresh latency p99 > 5s И K≥2 → доля `claims_exhausted > 0` для горячего credential; хвост резолва = до 25s (5×5s), а не «coalesced». §17 DoD меряет coalescing, но НЕ меряет p99 резолва под `refresh_latency > backoff_cap`.

2. RefreshClaimRepo CAS contention — thundering herd на storage, не на IdP. Механизм: L1 защищает только IdP POST (один на реплику). Но `try_claim` (durable CAS) вызывается КАЖДОЙ проигравшей репликой на КАЖДОЙ из 5 попыток. При K реплик на популярный credential = до K×5 CAS-write/UPDATE в окно ~25s на одну строку. Это row-level write contention в Postgres (одна строка claim → сериализация транзакций, lock wait). Слом: K=50 реплик × 5 = 250 CAS на одну строку в 25s ≈ 10 write-CAS/s на single row — это hot-row, и при истечении пачки credential одновременно (см. #6) число строк не спасает: каждая горячая строка независимо generирует свой CAS-storm. Метрика: `claims_contended` rate и DB lock-wait на claim-table при K≥20. Design нигде не бюджетирует CAS QPS на claim-store; §22 row B заявляет «only one replica refreshes» — верно для IdP, ЛОЖНО для CAS-нагрузки.

3. Rotation fan-out O(F·S) не ограничен и не описан как scale-инвариант. Механизм: §22 explicitly выносит rotation fan-out / `on_credential_refresh` из credential-крейта («Resource fan-out — out of scope», §2). Но ротация одного shared credential (DB master cred, общий OAuth app) → для каждого тенанта F × каждого слота S, держащего guard на этот credential, нужен pool-swap/reconnect. Это O(F·S) reconnect-burst в момент ротации. Слом: F=1000 тенантов × S=3 слота = 3000 одновременных reconnect/pool-rebuild на DB при blue-green cutover → connection-storm на целевую БД (та же грабля, что HikariCP-#1836, который проектная память уже знает). DESIGN передаёт это в resource/engine и НЕ ставит throttle/staggered-fanout инвариант. Метрика: reconnect QPS на downstream при F·S > пары сотен.

4. OwnerScopedKey закрывает confused-deputy, но НЕ адресует N+1 на list. Механизм: §10 rule 9 заменяет `store.get(&str)` (resolver.rs:491 — сейчас именно raw `self.store.get(credential_id)`) на privately-constructed `OwnerScopedKey`. Хорошо для безопасности. Но для management list/CRUD (§9) owner-scoped query = `WHERE owner_id = ?` — это требует составного индекса `(owner_id, id)`, иначе full-scan per tenant. И резолв слотов воркфлоу с M credential-слотами = M отдельных `get(OwnerScopedKey)` (N+1), нет batch `get_many(owner, [ids])`. Слом: воркфлоу с 20 credential-слотами × 100 параллельных исполнений = 2000 single-row roundtrip на dispatch. Метрика: store roundtrips на dispatch растёт линейно по слотам; нет index-инварианта в §17 DoD ни batch-API в порту.

Но нет числового бюджета (timeout=?, TTL=?, cache-size=?), нет указания, что cache keyed `owner×scheme×provider-fingerprint` (это упомянуто только для L1, §21 п.7, НЕ для ExternalProvider). Но jitter (resolver.rs:218 `rand::random_range(0..policy.jitter)`) размазывает только МОМЕНТ входа в refresh-окно, не число IdP-вызовов. При массовом одновременном истечении (тысячи токенов с TTL, выданных в одном batch-импорте → expiry в одну минуту) global semaphore=32 (l1.rs:200) — это per-replica потолок 32 concurrent IdP POST. Механизм: l1.rs:203 cap=4096 per-credential breakers; resolver.rs:234 при open circuit отдаёт stale-but-valid внутри early-window. А по факту кода (resolver.rs:209 routes на `state.expires_at()`+const `REFRESH_POLICY`, policy() с нуля consumers, hardcode `C::KEY != OAuth2Credential::KEY` на 536) Nebula сегодня ПОЗАДИ n8n: у n8n есть одна работающая data-driven функция, у Nebula — две рассинхронизируемые правды (Capabilities bitflag + policy) и ни одной, которую вызывает resolver. Moat ломается при метрике: 0 production-consumers у policy() = чистый паритет-на-бумаге, пока §22 строки A–I не станут зелёными тестами. До этого момента «moat» = отрицательный.

2. Reactive-only-в-1.0 — это конкурентный гандикап, выданный за принципиальность. n8n и Windmill рефрешат проактивно (по expires_in/TTL). Nebula отложила проактив в 1.1 (ADR-0084), и при этом lease/scheduler.rs УЖЕ рефрешит на N% TTL (Finding 5: «reactive-only already half-violated»). Значит «reactive-only» — не архитектурная позиция, а недоделанная: первая же нагрузка, где у credential нет inline `expires_at` (Vault-lease, k8s SA token, client-credentials без refresh_token), проваливается в silent-never-refresh (Finding 2). Метрика слома: любой провайдер с server-tracked TTL (expires_at: None) → 0 рефрешей до истечения → 503 на следующем вызове. n8n такие кейсы тащит.

3. «Один крейт = AI-discoverability» — это позиционирование под собственный инструментинг, не под пользователя/конкурента. AWS SDK держит `aws-credential-types` отдельно от runtime ИМЕННО чтобы трогать контракт без перекомпиляции 6k LOC (Finding 5). Nebula жертвует compile-firewall ради «агент читает в одном месте» и компенсирует это `#[cfg(test)]` arch-test'ом — то есть discipline-gate вместо type/crate-gate, что прямо противоречит собственному feedback_type_enforce_not_discipline. Ломается при метрике: 5-фазный contract-rewrite, который чаще всего трогает contract/, на каждый чих пересобирает runtime; на CI/инкрементале это стоимость, которой у конкурентов нет. Это не moat, это самоналоженный налог.

4. OAuth Plane Law (A vs B, zero HTTP в credential) — реально аккуратнее n8n, но это НЕ дифференциатор для пользователя, это внутренняя гигиена. Конкурентно невидимо: ни один потенциальный клиент не выбирает движок по «у них credential-крейт не монтирует роуты». А цена — узкий `RefreshTransport` seam, который сам DESIGN (rule 8, Finding 3) признаёт сегодня `SHOULD`, не enforced: второй composition root (CLI/worker/test) инжектит permissive transport и обходит SSRF. То есть заявленный invariant пока косметика. Moat-claim здесь = инженерная опрятность, выданная за рыночное отличие.

5. «Code-per-protocol / config-per-provider» — это буквально n8n `extends ['oAuth2Api']` + provider-data, Airflow `provider.yaml conn-fields`, Vault `FieldSchema`, AWS `ImdsProvider`+config (Finding 4: «every seat ships provider-as-data»). Nebula переименовала общеотраслевой паттерн в «OAuth2Protocol + OAuth2ProviderConfig» и при этом НЕ реализовала (zero grep hits; oauth2.rs = 1481 LOC монолит, проверено). Изобретение велосипеда хуже существующего: у конкурентов это работает, у Nebula это spec. Ломается при метрике: 2-й и 3-й SaaS-провайдер (нужно реально добавить github+slack без нового Rust-типа) — сегодня невозможно, потому что `provider()` не существует.

6. CredentialGuard<Scheme> bind-to-output + OwnerScopedKey — единственный участок, где есть НАСТОЯЩИЙ структурный отрыв (compile-time scheme-typing + owner-isolation-by-type, no `get(&str)`). Но он в DESIGN как target, а в коде confused-deputy открыт (Finding 3.1: `binding.fingerprint() == from_scope(scope)` — значение сравнивается с собой, store грузит по raw id). То есть самый сильный потенциальный moat сейчас — дыра безопасности. Метрика слома: один tenant с валидным credential_id чужого tenant → resolve проходит, потому что owner-gate отсутствует на store-уровне.

7. Стратегический промах позиционирования: DESIGN/CONFERENCE построены как «победить n8n/Windmill/Airflow по их же багам» (§22 — 9 их GitHub-issue превращены в regression-контракты). Это догоняющая, а не moat-стратегия: лучший мыслимый исход — «у нас те же фичи, но без их 9 известных багов». Это создаёт паритет+качество, не отрыв. Настоящий moat (per memory project_competitive_positioning) — это credential-as-engine-primitive ВНУТРИ durable workflow-движка с типизированными ресурсами: ни n8n, ни Windmill не дают типобезопасный slot-binding credential→resource→action через DAG. Этого в DESIGN почти нет как нарратива — он утоплен в OAuth-механике.

СЦЕНАРИИ

1. Vault-lease silent-never-renew (Finding 2). Шаги: зарегистрировать credential со `RefreshStrategy::Lease`, expires_at: None (server tracks TTL); под нагрузкой 100 req/s long-running action держит guard 20 мин; TTL=15 мин. resolver.rs:209 `state.expires_at().is_some_and(...)` = false → refresh-окно никогда не входится, `is_auto_renewable()` не зовётся → на 15-й минуте все вызовы 401/503. n8n с server-side TTL такой кейс ведёт. Нагрузка слома: 1 credential, любой TTL < длительности экзекуции.

2. Concurrent refresh storm на rotated refresh_token (§22 B, n8n #12742). Шаги: 50 параллельных нод на одном OAuth2 credential с one-time refresh_token; токен в окне early_refresh. L1 coalescer есть per-(owner,id), НО при non-parseable legacy id (resolver.rs:265) падает в `refresh_via_l1_only` без L2 claim → на 2 репликах два winner'а сжигают один refresh_token → `invalid_grant` → reconnect-loop. Нагрузка слома: 2 реплики × 1 credential с legacy id + rotation провайдер (GitHub/Revolut #4582).

3. Cross-tenant credential read через confused-deputy (Finding 3.1). Шаги: tenant B знает credential_id tenant A (утёк в логе/URL); B вызывает action со `slot_binding` на этот id; `resolve_for_slot` проверяет `fingerprint == fingerprint` (всегда true), `resolver.resolve(id)` грузит по raw id без owner-gate → B получает Scheme от credential A. Нагрузка слома: 1 запрос, 1 утёкший id. Это прод-блокер, не масштаб.

4. Secrets-backend outage стопорит весь DAG-parse (§22 H, Airflow). Шаги: ExternalProvider указывает на cloud SM с p99=8s; 500 нод на старте резолвят credential на hot path без timeout/cache → активация воркфлоу блокируется на 500×8s. DESIGN заявляет ordered chain+timeout+cache+fail-closed как target (rule 17), кода нет. Нагрузка слома: 1 медленный backend + ≥10 одновременных активаций.

5. Provider rotates refresh_token, не персистится (§22 A, n8n #30345). Шаги: refresh успешен, провайдер вернул новый refresh_token; CAS-write пишет частичный State без нового токена → через 1 час `invalid_grant` навсегда. rule 13 это адресует на бумаге; пока perform_refresh (resolver.rs:521) идёт через `try_oauth2_refresh` с serde round-trip — риск потери поля при schema-drift реален. Нагрузка слома: 1 credential, 1 ротация, провайдер с rotating tokens.

ЧТО ДОБАВИТЬ В DESIGN

1. §22-инвариант J «Lease/server-TTL credential обязан рефрешиться без inline expires_at»: resolver.rs запрещено гейтить только на `state.expires_at()`; DoD Phase-1 = тест «RefreshStrategy::Lease с expires_at:None входит в refresh-окно через policy(&State)/is_auto_renewable». Это закрывает silent-категорийный провал (Finding 2) и снимает паритетный отрыв n8n.

2. §17 Phase-1 DoD-инвариант: «policy() имеет ≥1 production-consumer и `C::KEY != OAuth2Credential::KEY` branch удалён», с arch-тестом, который грепом падает на наличие любого hardcoded `*Credential::KEY` сравнения в runtime/. Без этой метрики «policy-first routing» остаётся vaporware, а moat-claim — ложным.

3. Новый §-блок «Competitive moat thesis (explicit)»: вынести типизированный credential→resource→action slot-binding через TypedDAG как ЗАЯВЛЕННЫЙ отрыв (то, чего нет у n8n/Windmill), и для каждого пункта — что именно конкурент НЕ может выразить. Сейчас moat растворён в OAuth-механике; нужен абзац «почему это не догоняющая стратегия» с фальсифицируемым критерием (напр. «slot-binding type-mismatch = compile error, у n8n = runtime»).

4. §17-инвариант на legacy-id путь: «non-parseable credential_id запрещён в multi-replica проде» (resolver.rs:265 `refresh_via_l1_only` без L2 = сценарий 2). DoD = либо L2-claim обязателен для всех id, либо legacy-fallback `#[cfg(test)]`-only. Иначе concurrent-refresh-storm воспроизводится по построению.

5. §10-инвариант ужесточить rule 8 с SHOULD→MUST по факту кода: «RefreshTransport seam — нет публичного конструктора, принимающего host-фильтр опционально; SSRF connect-resolver не отключаем ни из одного composition root», + arch-тест на отсутствие второго пути инстанцирования transport. Пока seam косметический, Plane Law — это документ, не инвариант.

Чистый вывод: из 6 «дифференциаторов» 4 — переименованный отраслевой паттерн (active-lifecycle, code-per-protocol, reactive-refresh, Plane-Law-как-рыночное-отличие), 1 — самоналог (single-crate), и только CredentialGuard<Scheme>+OwnerScopedKey — настоящий структурный отрыв, который сегодня дырявый. Удваивать стоит ИМЕННО его (type-level owner-isolation + scheme-bound guard + typed slot-binding через DAG), а OAuth-механику — догонять до паритета молча, не продавая как moat.
