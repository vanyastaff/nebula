# Key Types

`nebula-core::keys` provides validated, normalized string identifiers for plugin types,
parameters, and credentials. Unlike the UUID-based IDs in `id.rs`, these are human-readable
slugs with strict formatting rules.

---

## PluginKey

The primary identifier for an action plugin type. Normalized at construction time.

```rust
pub struct PluginKey(String); // inner is always normalized
```

### Normalization Rules

1. Leading/trailing whitespace is trimmed.
2. The string is lowercased.
3. Whitespace characters and hyphens are replaced with `_`.
4. Consecutive underscores are collapsed to one.
5. Leading and trailing underscores are stripped.

```rust
"HTTP Request"    → "http_request"
" My--Cool  Plugin " → "my_cool_plugin"
"a___b"           → "a_b"
"___hello___"     → "hello"
```

### Validation Constraints

After normalization the key must:
- Be non-empty (not just whitespace/underscores).
- Contain only `a-z` and `_` (no digits, no uppercase after lowercasing).
- Be at most **64 characters** long.

### Construction

```rust
use nebula_core::PluginKey;

// Parse from string (infallible after normalization)
let key: PluginKey = "HTTP Request".parse()?;
let key = PluginKey::new("slack")?;
let key = PluginKey::try_from("my-plugin")?;
```

### Errors

```rust
pub enum PluginKeyError {
    Empty,              // input was empty or only whitespace/underscores
    InvalidCharacters,  // normalized key contains non [a-z_] characters
    TooLong,            // normalized key > 64 chars
}
```

### Traits

| Trait | Behavior |
|---|---|
| `Display` | Renders the normalized string |
| `FromStr` | Normalizes and validates |
| `Serialize` | JSON string of normalized key |
| `Deserialize` | Normalizes on deserialization — rejects invalid |
| `PartialEq<str>` | Compares to raw `&str` after normalization |
| `Hash` | Based on normalized string |

### Usage

```rust
// In Action trait (nebula-action):
fn metadata(&self) -> ActionMetadata {
    ActionMetadata {
        plugin_key: "http_request".parse().unwrap(),
        // …
    }
}

// Registry lookup:
let action = registry.get(&"HTTP Request".parse::<PluginKey>()?);
// Works because normalization is idempotent: "http_request" == "HTTP Request" after parse
```

## ParameterKey and CredentialKey

Domain-namespaced keys generated by `domain_key::key_type!`:

```rust
key_type!(ParameterKey, PrameterDomain);   // domain: "parameter"
key_type!(CredentialKey, CredentialDomain); // domain: "credential"
```

These are type-safe wrappers that prevent passing a `CredentialKey` where a
`ParameterKey` is expected. Both are `Clone`, `Hash`, `Eq`, `Serialize/Deserialize`.

Re-exported from `nebula-core` for use in `nebula-parameter` and `nebula-credential`.

---

## UuidParseError

```rust
pub use domain_key::KeyParseError;
// also exported from prelude as `UuidParseError` (alias)
```

Returned when parsing a UUID-based ID from a string fails.

---

## Design Note

`PluginKey` was introduced to allow plugin authors to register their action as
`"HTTP Request"` (human-friendly) while the registry stores and looks it up as
`"http_request"` (normalized). Serde deserialization also normalizes, so plugin
keys coming from JSON configuration are always in a canonical form before any
comparison or lookup.
