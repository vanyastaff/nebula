## Verdict

**RATIFY-WITH-NITS.** §10.4 + §10.5 + §12.2 close the structural community-plugin migration story cleanly and address the CP1 R3 gap I flagged (`StatelessAction` is now explicitly named as the post-seal community-author target, surfaced in `nebula-sdk::prelude` per §9.4 line 1585, and discoverable through both §12.2's worked example and §10.4 step 5's T6 sub-bullet). Five non-blocking nits surface from the **plugin-author lens** specifically: (1) §10.4 step 1 omits the `::semver::Version` Cargo.toml gap that survived from Phase 1 CC1 — first compile after migration *will* fail and the guide doesn't preempt it; (2) §10.4 step 5 T2 doesn't tell the author to re-run codemod after adding the missing zone, even though §10.2.1 makes re-runs idempotent — a discoverable workflow gap; (3) §10.4 step 5 T2 doesn't tell the author *how to find* the missing credential type if the codemod can't infer it; (4) §10.4 + §12.2 use **inconsistent attribute syntax** for the unresolved `control_flow` form (`control_flow = ...` in §10.4 step 5 + §10.2 T6 row vs `control_flow,` flag form in §12.2 example) — a plugin author copy-pasting either example will hit the other form's parse rule; (5) §12.2 names `nebula-action::sealed_dx::*` as cascade-internal but the `nebula-sdk::prelude` re-export of the five DX trait *identifiers* (per §9.4 line 1589) means a plugin author CAN write `impl ControlAction for X` — the bound-shape import compiles and the seal trips at the impl statement, not the import. The §12.2 prose says "do not implement directly" but the prelude exposes the bound-shape import; §12.2 should clarify that the export is **bound-shape-only for type aliases**, not an authoring target.

Severity is RATIFY-WITH-NITS, not REVISE — the structural decisions (T1-T6 codemod scope, dual-mode AUTO+MANUAL, MIGRATION.md as cascade-landing artifact, `StatelessAction` direct-impl path) are sound; nits are localized doc edits or 1-line clarifications.

## §10.4 plugin author migration guide discoverability

### What works

**The 7-step ordering is what I would write down for a community plugin author.** Bump → dry-run → review diff → apply auto → resolve manual markers → run tests → update docs. Each step is a discrete commit-able unit; nothing is hidden inside another step. This matches the workflow a plugin author would naturally try — they don't have to invent the order.

**Step 5 sub-bullets are concrete per-transform.** T2 (no-key credential), T5 (`redacted_display!` wrap), T6 (ControlAction → StatelessAction) each get their own sub-bullet with the **target syntax named inline** — "replace with `ctx.resolved_scheme(&self.<slot_name>)?`", "rewrite `impl ControlAction for X` to `impl StatelessAction for X` + `#[action(control_flow = ...)]`". A plugin author resolving a `// TODO(action-cascade-codemod): manual rewrite required` marker at line 47 of their code can read the marker, jump to §10.4 step 5 T2 / T5 / T6, and see the target form without grepping the spec. This is the right level of indirection.

**Cost estimate is calibrated against measurable evidence.** §10.4 last paragraph: trivial plugin <30 min, complex plugin 2-4 hr. My Phase 1 measurement was 32 min to write **one** credential-bearing action **with no migration guide** — so 30 min for the *entire* migration of a 1-action / 1-credential plugin is consistent with the codemod absorbing the bulk of the busywork (T1 derive-rewrite, T3 Box→Arc, T4 HRTB collapse) and the author only resolving 1-2 manual markers. The 2-4 hr ceiling for complex plugins is also plausible: 5 actions × (30 min understanding the new shape + 15 min of manual marker resolution per credential pattern) ≈ 4 hr. Numbers are not aspirational.

**MIGRATION.md ships in `crates/action/` alongside the cascade-landing PR.** Per `feedback_active_dev_mode.md` ("DoD includes migration guide for breaking changes") — this is the right placement. A plugin author doing `cargo update -p nebula-action` and hitting compile errors will look in their dependency's source tree for `MIGRATION.md` first; it'll be there. Not a follow-up issue, not a wiki link, not docs.rs only. Right call.

### Nits

1. **`::semver::Version` Cargo.toml dep is not in step 1 and is not a codemod transform.** §10.4 step 1 says "Bump `nebula-action` and `nebula-sdk` to the new release." Phase 1 finding CC1 (and §4.6.2 line 920 confirms it survives CP3): `#[action]` macro emits `.with_version_full(::semver::Version::new(major, minor, patch))` with the absolute `::semver::` path. Plugin authors who don't already have `semver = "1"` in their Cargo.toml will hit `error[E0433]: cannot find 'semver' in the crate root` on first compile *after* the migration — and the migration guide doesn't preempt this. The codemod operates on `.rs` files (per §10.2.1), not `Cargo.toml`; T1 doesn't fix it. **NIT — RECOMMEND** — §10.4 step 1 should add: "If your plugin's `Cargo.toml` does not already depend on `semver = \"1\"` directly, add it. The `#[action]` macro emits `::semver::Version` paths; consumer crates need the dep declared at their own Cargo.toml level." OR — alternative architectural fix per Phase 1 CC1 — emit `::nebula_action::__private::semver::Version` (re-exported through nebula-action) so the consumer never needs the direct dep. That's a CP4 §15 housekeeping pick; for CP3 §10 the doc note is the smaller fix.

2. **Step 5 T2 missing "re-run codemod after declaring the zone".** §10.2.1 line 1681 confirms re-runs are idempotent, which means the workflow `add credentials(...) zone → re-run codemod → AUTO transform now applies` is supported. But §10.4 step 5 T2 says: "If the credential type is unknown at the call site, the plugin author must add an explicit `credentials(...)` zone to the action's `#[action(...)]` attribute first." It doesn't say *and then re-run the codemod, the AUTO path will pick up the change*. A plugin author reading step 5 T2 will assume they have to *also* hand-rewrite each `ctx.credential::<S>()` call site even after declaring the zone — duplicating effort. **NIT — RECOMMEND** — extend T2 sub-bullet: "...attribute first. **Then re-run `nebula-action-codemod --apply`; the previously-MANUAL T2 sites will now resolve to AUTO once the zone is declared, since the codemod can infer the slot from the matching credential type.**" Two sentences; saves a plugin author 15 min on a 5-action plugin.

3. **Step 5 T2 doesn't tell the author *how to find* the missing credential type.** "If the credential type is unknown at the call site" — but how does the author *discover* what `ctx.credential::<S>()` was supposed to resolve? Today's code uses type-name lowercase as the lookup key (Phase 1 finding 9), so a call like `ctx.credential::<SecretToken>()` looks up key `secrettoken`. Post-migration the author needs to translate that into a `credentials(slot: Type)` declaration — but they need to find which workflow node config / credential snapshot was historically resolved at that key. The guide doesn't say "grep the workflow JSON manifests for your action", "check `nebula-credential` registry", or "look at the type the engine was binding to." **NIT — RECOMMEND** — extend T2 sub-bullet with discovery hint: "To find the credential type historically bound at the call site: (a) check the workflow JSON / YAML manifests that reference your action's `key` for the bound credential type name; (b) if the call site has no explicit type annotation (`ctx.credential::<_>()` form), grep for `register_action(<your_action_key>, ...)` in the consuming workflows or the test harness — the bound type is named there." This converts a discovery puzzle into a 2-step grep; otherwise the author falls back to running the action against tests and watching the runtime `ResolveError::NotFound` to enumerate what's missing.

4. **§10.4 step 6 says "New compile-fail probes (§5.3 Probes 1-7) catch shape violations early"** — but Probes 1-7 are macro-crate-internal trybuild tests (per CP2 §5 + 09d review), they are **NOT** run automatically against community plugin code. A plugin author reading step 6 might assume `cargo test` in their plugin crate exercises the probes — it does not; the probes live in `crates/action/macros/tests/`. The author's tests catch their *own* shape violations only insofar as the plugin's tests instantiate the action and call into the dispatch path. **NIT** — step 6 should either drop the "Probes 1-7" reference or rephrase it as "The macro's internal compile-fail probe coverage means shape violations in your `#[action]` attributes surface as `compile_error!` at *your* crate's compile time with span pointing at the offending field — not as runtime errors." This is what the dual-enforcement (§4.4) actually delivers; the prose framing of "Probes 1-7 catch your bugs" overstates the harness's reach.

## §10.5 AUTO / MANUAL workload realism

### What works

**The 70/30 split is honest, not aspirational.** §10.5 last paragraph: "Automating those would be a `feedback_no_shims.md` violation." Right framing — the 30% is *not* "we ran out of time", it's "silently rewriting credential bounds and error sanitization is the worst-possible-UX category from Phase 1 finding #6." Plugin authors get a TODO marker; they own the rewrite; the marker is discoverable in their IDE and their `cargo build` output.

**MANUAL-REVIEW mode behavior is clear at §10.2.1.** "Codemod inserts `// TODO(action-cascade-codemod): ...` marker + leaves the original line unchanged; reviewer applies the rewrite by hand using the marker hint." A plugin author searching for `// TODO(action-cascade-codemod)` in their IDE finds every site at once — no risk of "did I miss one?" The marker is greppable, not buried in a JSON output file.

**T1/T3/T4 AUTO classification is correct against my Phase 1 evidence.** T1 (`#[derive(Action)]` → `#[action(...)]`) is mechanical attribute extraction; it has no semantic content beyond renaming, so AUTO is right. T3 (Box→Arc) and T4 (HRTB collapse) are pure-mechanical token rewrites against a syntactic pattern — the codemod doesn't have to understand semantics. AUTO holds.

**T2/T5/T6 MANUAL-REVIEW classification is correct against my Phase 1 evidence.** T2 needs to know *which credential type* the call site historically resolved (Phase 1 found this can be 8-source-file deep). T5 needs to know *whether the error's Display can leak credential material* — a domain judgment, not a syntactic pattern. T6 needs to preserve control-flow-specific behavior — the codemod can't tell `Continue { skip_if_empty: true }` from `Continue { skip_if_empty: false }` from a generic pattern. MANUAL-REVIEW is the only correct classification per `feedback_no_shims.md`.

### Nits

1. **§10.5 doesn't quantify the per-plugin manual-marker count.** §10.3 gives per-Nebula-consumer site counts (engine ~5 T2 sites, ~5-7 T5 sites; api ~1 T2; cli ~1-2 T2). But for a **community plugin** the per-plugin manual-marker count is the load-bearing input to "should I migrate now or wait?" — and the §10.4 cost estimate (<30min trivial / 2-4hr complex) leaves the author guessing how many T2/T5/T6 sites their specific plugin has until they run `--dry-run`. **NIT** — §10.5 add one sentence: "Per-plugin manual-marker count scales linearly with the number of `ctx.credential::<S>()` call sites (T2) + `tracing::error!(... = %e)` foreign-error sites (T5) + `impl ControlAction` blocks (T6). Run `nebula-action-codemod --dry-run --count-only` (CP4 implementation scope) for a quick estimate before committing to migrate." If `--count-only` is too aspirational for CP3 spec, drop it; the linear-scaling sentence alone is enough.

2. **§10.5 doesn't acknowledge the *interaction* between T1 (AUTO) and T2 (MANUAL).** A plugin that today writes `#[derive(Action)] struct X` + `#[nebula(credential = "slack_token")]` companion attribute (the pre-cascade form) hits T1 first (AUTO rewrites `#[derive(Action)]` to `#[action(...)]` and folds the companion `#[nebula(...)]` attributes into the new `#[action(...)]`) — but `credential = "slack_token"` is the **silent-drop string-form** that CP2 §4.7.2 hard-rejects. So T1 AUTO mode produces a `#[action(credential = "slack_token", ...)]` shape that **fails CP2 §4.7.2 hard-error at compile time**. The author then has to manually rewrite to `credentials(slack: SlackToken)`. This is a T1+T2 chain, but §10.2 / §10.5 treats T1 as pure-AUTO. **NIT — RECOMMEND** — §10.2 T1 row notes column should add: "T1 AUTO transform of `#[nebula(credential = \"key\")]` companion produces a `compile_error!` per §4.7.2 at the rewritten attribute; author resolves by promoting to T2 MANUAL flow (declare `credentials(slot: Type)` zone, then re-apply codemod)." This makes the T1→T2 escalation visible up front instead of surprising the author after T1's diff looks clean. Alternatively, T1 could detect the `credential = "string"` case during the rewrite and emit the TODO marker directly without producing the broken intermediate — but that's a codemod implementation detail, not a §10.2 contract change.

## §12.2 community plugin DX clarity

### What works

**Structural fix: `StatelessAction` is named explicitly as the community-plugin direct-impl target.** Per §12.2 line 1885: `impl StatelessAction for ControlExampleAction` — and the worked example shows the **complete** authoring shape (struct + `#[action(...)]` attribute + `impl StatelessAction { type Input/Output/Error; fn execute<'a>(...) -> impl Future<...> + Send + 'a }`). This **closes my CP1 R3 gap** — at CP1 §2.6 the migration target was implicit; at CP3 §12.2 it's surfaced with a 30-line copy-pasteable example. A plugin author hitting `error: trait ControlAction is sealed` follows §10.4 step 5 T6 → reads §12.2 → has the answer.

**§9.4 prelude line 1585 + §12.2 line 1875 are consistent.** `nebula-sdk::prelude::*` exposes the four primary traits; `use nebula_sdk::prelude::*;` is the canonical first-line import in §12.2's example. Phase 1 finding 11 (10 trait surfaces in `lib.rs` confused authors) is closed: the plugin author's prelude is **scoped to authoring entry points**, the lower-level `Handler` / `Adapter` / `SlotBinding` types live behind `use nebula_action::*` (§9.4 line 1592). One namespace per role.

**"Plugin author never names `ControlAction` trait or `ControlActionAdapter` type" is the load-bearing clarity.** §12.2 line 1904. This is the right framing — sealed-DX is a contract-shape, not an API surface, for community authors. The macro emits the adapter; the plugin author owns the typed body. Architecturally clean.

**Codemod-covered migration eliminates the "I have an existing ControlAction impl, what now?" panic case.** §12.4 + §10.2 T6 + §10.4 step 5 T6 form a complete chain: existing `impl ControlAction for X` is detected by the codemod, rewritten to `impl StatelessAction for X` + `#[action(control_flow = ...)]` for the trivial pass-through case, MANUAL-REVIEW for the edge cases (custom `Continue`/`Skip`/`Retry` reasons). Plugin author with one mechanical ControlAction → 5 minutes of codemod review. Plugin author with intricate control-flow logic → MANUAL marker, hand-port. Either way they're never stranded.

### Nits

1. **§12.2 worked example uses `control_flow,` (flag form) but §10.4 step 5 T6 + §10.2 row T6 use `control_flow = ...` (key=value form).** §12.2 line 1880 attribute zone: `control_flow,` (line 1880 has just `control_flow,`). §10.4 step 5 T6 sub-bullet: `#[action(control_flow = ...)]`. §10.2 T6 row: `#[action(control_flow = ...)]`. Same spec, two different syntactic forms for the **same unresolved attribute**. Per §12.2 line 1904 explicitly: "The exact attribute zone syntax — `control_flow` flag vs `control_flow = SomeStrategy` config — is CP4 §15 scope." **The spec acknowledges the ambiguity and uses both forms in different examples without picking a placeholder convention.** A plugin author copy-pasting §12.2's example will write `control_flow,` and the codemod's T6 output (using `control_flow = ...`) will not match. **NIT — REQUIRED EDIT** — pick one placeholder convention for CP3 freeze and use it consistently across §10.2 / §10.4 / §12.2. Suggest `#[action(control_flow = …)]` (key=value with elided ellipsis) so it's clear something goes there without committing to the exact value shape; OR write `#[action(control_flow /* CP4 §15 picks: flag or key=value */)]` to make the ambiguity explicit at every example. Either is fine; mixed forms in the same spec is the trap. Actual spelling decision is CP4 — but freeze invariant 3 (per §0.2) makes "internally inconsistent placeholders" an editorial bug worth fixing now.

2. **§12.2 doesn't address the "I imported `ControlAction` from prelude, why can't I impl it?" trap.** §9.4 line 1589: prelude exposes "the five DX trait identifiers (`ControlAction`, `PaginatedAction`, `BatchAction`, `WebhookAction`, `PollAction`) — bound-shape only; community plugins do not impl directly per §9.2 / ADR-0038." Bound-shape-only means the trait *identifier* is in scope (so `Capability::Control` / `where T: ControlAction` works for type-bounds elsewhere in the engine) but the seal trips the *impl* statement. A plugin author who reads `use nebula_sdk::prelude::*;` in §12.2's first line and then writes `impl ControlAction for X { ... }` will get... what diagnostic? The sealed-trait error message (the `private_in_public` family or `cannot implement sealed trait`). §12.2 doesn't tell the author what error they'll see, doesn't tell them the prelude import is bound-shape-only, and doesn't anticipate the question. **NIT — RECOMMEND** — §12.2 add a paragraph after line 1904: "**Why is `ControlAction` in `nebula-sdk::prelude` if I cannot implement it?** The DX trait identifiers (`ControlAction`, `PaginatedAction`, etc.) are exported as **bound-shape symbols** for engine-side type bounds (`where T: ControlAction`) and rustdoc cross-references, **not as authoring targets**. Attempting `impl ControlAction for X` produces a sealed-trait compile error pointing at the impl statement; the fix is in §10.4 step 5 T6 (move to `impl StatelessAction for X` + `#[action(control_flow = …)]`)." This converts a startled plugin author into a directed plugin author; the diff is one paragraph.

3. **§12.2 doesn't cite the `nebula-action-codemod --check` flag or any "is my plugin already migrated?" command.** A plugin author bumping to a new release has two states: (a) "I migrated last release; my code already conforms"; (b) "I just bumped; I need to migrate." Today, the only way to disambiguate is `cargo build` (errors → migrate; clean → done). The codemod could provide `nebula-action-codemod --check` (no-op return code 0 if green, exit 1 + listing if any pre-cascade patterns remain) — useful for CI integration. §10.2.1 names `--dry-run` and `--apply`; §12.2 doesn't reference any check-mode. **NIT** — flag for §10.2.1 CP4 implementation scope; CP3 §10/§12 don't have to commit the `--check` flag, but the **idea of a "is my plugin clean?" command is a discoverability win** that's worth a forward-pointer in §12.2 close: "Future codemod versions may add `--check` mode for CI integration; CP4 §15 picks." Cheap; sets the expectation.
