# Poll trigger ‚Äî deferred API-v2 changes

**Status:** skeleton / deferred
**Scope:** `crates/action/src/poll.rs`, `crates/action/tests/dx_poll.rs`,

**Blocked by:** nothing technical; waiting on priority + runtime capability

## Purpose

Park the breaking / cross-crate / policy-re-homing changes that came
out of the poll audit (sessions 2026-04-13) so they do not rot. Each
item here has a concrete motivation from the audit or from one of the
three agents (architect, tech-lead, sdk-user). None of them is
speculative; all of them require either an API break or runtime
support we do not yet have.

This file is a **skeleton**. When we decide to do one of these items,
expand it into a proper implementation plan or split it into its own
`YYYY-MM-DD-poll-...md` file. Do not drive code directly from this
doc.

## Items

### V1 ‚Äî `PollBudget` replacing advisory `max_pages_hint`

**Problem.** Today `max_pages_hint` is a `Option<usize>` on
`PollConfig` that the adapter does not enforce. The action author is
supposed to read it inside `poll()` and return `PollResult::partial`
when the limit is reached. Sdk-user review: *"a config field the
config doesn't act on"*. Nobody will remember to honour it under
deadline pressure.

**Proposal.** Introduce:

```rust
pub struct PollBudget {
    deadline: Instant,            // poll_timeout from config
    max_pages: Option<usize>,     // from config
    max_events: Option<usize>,    // new
}

impl PollBudget {
    pub fn expired(&self) -> bool;
    pub fn pages_remaining(&self) -> Option<usize>;
    pub fn events_remaining(&self) -> Option<usize>;
    pub fn tick_page(&mut self);
    pub fn tick_events(&mut self, n: usize);
}
```

`PollAction::poll` signature becomes:

```rust
async fn poll(
    &self,
    cursor: &mut PollCursor<Self::Cursor>,
    budget: &mut PollBudget,
    ctx: &TriggerContext,
) -> Result<PollResult<Self::Event>, ActionError>;
```

Action authors check `budget.expired()` inside their page loops and
return `PollResult::partial` cooperatively. This also solves the
cancellation-safety issue flagged in the audit (#9): `poll_timeout`
becomes cooperative instead of grafting `tokio::time::timeout` on top.

**Breaking.** Yes ‚Äî `PollAction::poll` signature changes. All
implementors need the new argument.

**Cost.** ~1 day. Touches the trait, the adapter loop, all existing
PollActions (`examples/poll_habr` only, in-tree).

### V2 ‚Äî Drop `Default` bound on `PollAction::Cursor`

**Problem.** `type Cursor: Serialize + DeserializeOwned + Clone + Default + Send + Sync`.
The `Default` bound is only used by the trait's default
`initial_cursor` implementation (`Self::Cursor::default()`).
Actions that override `initial_cursor` don't need `Default` at all,
and "start from beginning of time" is almost never what a real
integration wants.

**Proposal.** Drop `Default` bound. Make `initial_cursor` a required
method with no default, OR provide a default that returns an error
demanding the user override it. Prefer the former ‚Äî forces authors
to make a deliberate choice between "start from now" and "start
from beginning".

**Breaking.** Yes ‚Äî every PollAction must either gain an
`initial_cursor` impl or accept the removal of the default.

**Cost.** ~2 hours. In-tree impact: `examples/poll_habr` needs one
`initial_cursor` method.

### V3 ‚Äî Re-home `jitter` and `backoff_factor` to runtime policy

**Problem.** Sdk-user: *"I have zero information about how many other
triggers share my schedule; the runtime does."* Action authors don't
know what jitter value is appropriate for their fleet density or
how aggressive backoff should be when upstream is stressed.

**Proposal.** Remove `jitter` and `backoff_factor` from `PollConfig`.
Adapter reads them from a runtime-provided policy object (e.g.
`ctx.scheduler_policy`) or uses sane defaults (10% jitter, 2x
backoff). Action authors declare only *characterisable* behavior:
*"I support idle backoff"* (bool) and the runtime picks the curve.

**Breaking.** Yes ‚Äî `PollConfig` loses fields. Constructors change
signature: `with_backoff(base, max)` instead of
`with_backoff(base, max, factor)`.

**Cost.** ~1 day. Also requires adding the policy injection path
through `TriggerContext`, which touches `nebula-runtime`.

### V4 ‚Äî Re-home `emit_failure` to workflow binding

**Problem.** Architect review: *"This is dispatch semantics ‚Äî at-
least-once vs best-effort vs fail-stop. It belongs on the trigger
binding, not the action. A payment integration isn't always
`RetryBatch`; it depends on the downstream."*

**Proposal.** Remove `emit_failure` from `PollConfig`. Move it to the
workflow-level trigger binding (i.e. the config that connects a
`PollAction` to a specific workflow instance). Runtime passes the
per-binding policy into `TriggerContext` (new field
`dispatch_policy: EmitFailurePolicy`).

**Breaking.** Yes, and this one also touches how workflows are
configured. Larger blast radius.

**Cost.** ~2 days. Touches workflow config shape in `nebula-core` or
`nebula-workflow`.

### V5 ‚Äî Cursor persistence (F8)

**Problem.** Already documented in `PollAction` trait docstring and
in `.project/context/active-work.md`. Process restart loses cursor
progress silently. Dangerous for payment/audit integrations.

**Proposal.** `TriggerContext` grows an optional
`trigger_state: Option<Arc<dyn TriggerStateStore>>` capability. The
adapter calls `load` at start, `store` after every successful
cycle (or on every checkpoint, configurable). Actions that want
persistence declare a `State` type (which may be `()` for the
in-memory case). `nebula-runtime` provides a real implementation
backed by the same storage the execution engine uses.

**Breaking.** Probably not ‚Äî can be additive if we keep the current
in-memory path as the default with no `TriggerStateStore` injected.

**Cost.** ~3‚Äì5 days. Cross-crate (runtime + storage + action).
**Blocked on:** runtime having real storage at all ‚Äî today only
`MemoryStorage` exists.

### V6 ‚Äî Delete `AtomicBool started` + `StartedGuard`

**Problem.** Architect review: *"runtime is the only caller, guard is
a footgun for stop‚Üístart deadlock, the guard is protection against
runtime bugs at the cost of legitimate restart patterns"*.

**Proposal.** Runtime takes ownership of the spawned task handle.
`TriggerHandler::start` returns something like a `TaskHandle` (or
the runtime spawns internally). `stop()` awaits the handle.
`AtomicBool` disappears because single-ownership is structurally
enforced.

**Breaking.** Yes ‚Äî `TriggerHandler` signature changes, affects
webhook adapter too.

**Cost.** ~2 days. Requires webhook adapter refactor in parallel
since `TriggerHandler` is shared.

### V7 ‚Äî QueueTrigger / StreamTrigger trait families

**Tech-lead's call:** NOT until a real user asks. One universal
`TriggerHandler` with two DX specializations (poll + webhook) covers
90 % of SaaS integrations. Kafka/SQS/RabbitMQ/WebSocket belong in
their own families but introducing speculative abstractions before
anyone needs them rots. Leave this as a note in `active-work.md`
under "Possible future trigger families".

**Cost when triggered:** ~2 weeks RFC + implementation per family.
Not scoped here.

## Prioritisation order when we pick this up

1. **V2** ‚Äî cheapest, clearest win, unblocks cleaner `initial_cursor`
   contract. Do first.
2. **V1 (PollBudget)** ‚Äî mid cost, high value. Fixes cancellation
   safety AND turns `max_pages_hint` from lie into enforced budget.
3. **V6** ‚Äî requires runtime ownership model. Do together with any
   work that touches runtime's trigger lifecycle.
4. **V3** ‚Äî depends on `SchedulerPolicy` capability on
   `TriggerContext`. Can be done after V6 lands.
5. **V4** ‚Äî workflow-level config change, larger blast radius. Do
   when we have a reason (first payment integration shipping).
6. **V5** ‚Äî blocked on runtime storage. Revisit when Postgres
   storage lands.
7. **V7** ‚Äî not until a real Kafka user shows up.
