**Repo:** https://github.com/temporalio/sdk-rust

## 0. Project Metadata

The repository is **dual-purpose**: it is simultaneously the Rust language SDK (user-facing), and the shared `sdk-core` engine used by multiple language SDKs. This is architecturally critical — the same codebase powers TypeScript, Python, .NET, Ruby, and Rust SDKs. The "Core SDK" layer communicates with the Temporal server via gRPC and handles all state machine logic. The "Lang SDK" layer (Rust: `temporalio-sdk`) sits on top and provides the ergonomic Rust API.

---

## 1. Concept Positioning [A1, A13, A20]

**Author's sentence (from README):** "Core SDK that can be used as a base for other Temporal SDKs."

**Operational sentence (from code):** A Rust library that connects to a Temporal Server over gRPC, drives workflow state machines via history replay, dispatches workflow/activity/nexus tasks to user-defined async Rust functions, and reports completions back to the server.

**Comparison with Nebula:** Temporal SDK is a *client* of Temporal Server; it does not own persistence, scheduling, or the DAG itself. Nebula is a *self-contained* engine that owns all of these. This is the fundamental architectural contrast: Nebula = engine. Temporal SDK = stateless worker client. Temporal's durability is server-side; Nebula's durability is application-side (frontier-based checkpoint, frontier scheduler).

## 3. Core Abstractions [A3, A17] — DEEP

### A3.1 Trait shape

Temporal uses **two top-level kinds of work**: **Workflows** and **Activities** (plus Nexus operations in v0.3). There is no sealed trait or "5 kinds" enumeration. Instead, the design uses **associated-type marker traits**.

**WorkflowDefinition** (`crates/common/src/workflow_definition.rs:7`):
```rust
pub trait WorkflowDefinition {
    type Input: TemporalDeserializable + TemporalSerializable + 'static;
    type Output: TemporalDeserializable + TemporalSerializable + 'static;
    fn name(&self) -> &str;
}
```
This is implemented on **marker structs** generated by `#[workflow_methods]` macro — not on the workflow struct itself.

**ActivityDefinition** (`crates/common/src/activity_definition.rs:10`):
```rust
pub trait ActivityDefinition {
    type Input: TemporalDeserializable + TemporalSerializable + 'static;
    type Output: TemporalDeserializable + TemporalSerializable + 'static;
    fn name() -> &'static str where Self: Sized;
}
```

**WorkflowImplementation** (the impl-side trait) (`crates/sdk/src/workflows.rs:144`):
```rust
pub trait WorkflowImplementation: Sized + 'static {
    type Run: WorkflowDefinition;
    const HAS_INIT: bool;
    const INIT_TAKES_INPUT: bool;
    fn name() -> &'static str;
    fn init(ctx: WorkflowContextView, input: Option<...>) -> Self;
    fn run(ctx: WorkflowContext<Self>, input: Option<...>) -> LocalBoxFuture<'static, ...>;
    fn dispatch_update(...) -> Option<LocalBoxFuture<...>> { None }
    fn validate_update(...) -> Option<Result<(), WorkflowError>> { None }
    fn dispatch_signal(...) -> Option<LocalBoxFuture<...>> { None }
    fn dispatch_query(...) -> Option<Result<Payload, WorkflowError>> { None }
}
```
This is NOT sealed (no external crate restriction). It is `dyn`-compatible via `DynWorkflowExecution` type erasure (`crates/sdk/src/workflows.rs:424`). No GATs used. No HRTBs. The `WorkflowContext<W>` is generic over the workflow struct, which effectively encodes a weak typestate at the workflow struct level.

**DynWorkflowExecution** (`crates/sdk/src/workflows.rs:424`):
```rust
pub(crate) trait DynWorkflowExecution {
    fn poll_run(&mut self, cx: &mut TaskContext<'_>) -> Poll<...>;
    fn validate_update(&self, name: &str, data: &DispatchData) -> Option<Result<(), WorkflowError>>;
    fn start_update(...) -> Option<LocalBoxFuture<'static, ...>>;
    fn dispatch_signal(...) -> Option<LocalBoxFuture<'static, ...>>;
    fn dispatch_query(&self, ...) -> Option<Result<Payload, WorkflowError>>;
}
```
This is the `Box<dyn ...>` erased form stored in the worker's workflow table.

**StateMachine** (core FSM base) (`crates/common/src/fsm_trait.rs:10`):
```rust
pub trait StateMachine: Sized {
    type Error: Error;
    type State;
    type SharedState;
    type Event;
    type Command;
    fn on_event(&mut self, event: Self::Event) -> Result<Vec<Self::Command>, MachineError<Self::Error>>;
    fn name(&self) -> &str;
    fn state(&self) -> &Self::State;
    fn shared_state(&self) -> &Self::SharedState;
    fn has_reached_final_state(&self) -> bool;
    fn from_parts(state: Self::State, shared: Self::SharedState) -> Self;
}
```
This is the underlying trait for all 16+ command-level state machines (ActivityMachine, TimerMachine, ChildWorkflowMachine, UpdateMachine, NexusOperationMachine, etc.) found in `crates/sdk-core/src/worker/workflow/machines/`.

### A3.2 I/O shape

Workflow/activity I/O uses **Temporal Payloads** — protobuf-encoded byte arrays with an encoding metadata header. The `TemporalSerializable`/`TemporalDeserializable` traits gate I/O types. The default implementation uses `serde_json` (JSON encoding). Users can plug in custom `PayloadConverter` implementations (e.g., protobuf binary encoding — see issue #1142). There is no `Box<dyn Any>` or `serde_json::Value` at the API boundary; types must implement the conversion traits. Streaming output does not exist — all outputs are single Payload returns.

### A3.3 Versioning

Temporal uses the **`workflow.get_version()` / `ctx.patched()`** pattern for workflow versioning. This is not a trait-level distinction — the same workflow struct handles all versions. Versioning is **runtime branching** via `PatchMachine` (`crates/sdk-core/src/worker/workflow/machines/patch_state_machine.rs`). Workflows are referenced by **type name** (a `&'static str` generated by the macro or provided via `#[workflow(name = "...")]`). No compile-time v1/v2 distinction exists; no `#[deprecated]` on workflow types.

### A3.4 Lifecycle hooks

**Activities:** The async fn annotated `#[activity]` is the entire lifecycle. There is no pre/post/cleanup hook. Cancellation is checked via `ctx.cancelled().await` or `ctx.is_cancelled()` at `crates/sdk/src/activities.rs:153`. No idempotency key hook exposed at the trait level (idempotency is the activity author's responsibility).

**Workflows:** `#[init]` is an optional constructor hook. `#[run]` is the main body. `#[signal]`, `#[query]`, `#[update]` are dispatched handlers. No explicit on-failure hook at the trait level; panics are caught via `AssertUnwindSafe` in `workflow_future.rs`.

### A3.5 Resource and credential dependencies

**Explicit answer:** There is no mechanism to declare "this activity needs DB pool X or credential Y" at the trait level. Activities receive a `WorkflowContext` / `ActivityContext` and any additional dependencies must be provided via closure capture (e.g., `Arc<MyDb>` captured in the `#[activities]` impl struct). The macro system has no concept of resource injection. Compare Nebula where `A3.5` is addressed by resource scope annotations and the CredentialOps assoc type on actions.

### A3.6 Retry/resilience attachment

Activities declare `RetryPolicy` as part of `ActivityOptions` when *scheduled from workflow code* (`crates/sdk/src/workflow_context/options.rs`). The policy includes `initial_interval`, `backoff_coefficient`, `maximum_interval`, `maximum_attempts`, and `non_retryable_error_types`. **The server enforces retries**, not the SDK. The SDK's `ValidatedRetryPolicy` (`crates/sdk-core/src/retry_logic.rs`) is used for local activities only. There is no circuit breaker, bulkhead, or hedging at the SDK layer — those would be application-level patterns.

### A3.7 Authoring DX

Minimal example (~15 lines):
```rust
struct MyActivities;
#[activities]
impl MyActivities {
    #[activity]
    async fn echo(_ctx: ActivityContext, e: String) -> Result<String, ActivityError> {
        Ok(e)
    }
}
```
Proc macros handle registration boilerplate. The `#[workflow_methods]` macro (`crates/macros/src/lib.rs:52`) generates marker structs, trait impls, and factory registration. Authoring DX is good for activities; workflows require understanding `WorkflowContext<W>` and the deterministic select/join macros.

### A3.8 Metadata

Display name = workflow/activity type name (from `fn name() -> &'static str`). No i18n, no icon, no category, no rich metadata. Compare Nebula's planned metadata via `ActionMetadata`.

### A3.9 vs Nebula

Nebula has 5 action kinds (Process/Supply/Trigger/Event/Schedule). Temporal Rust SDK has **2 primary kinds** (Workflow, Activity) plus Nexus (partial). No sealed trait — the marker struct pattern is used instead. No GATs or HRTBs in the action trait hierarchy. Nebula's approach is more type-safe and more self-describing; Temporal's is simpler and relies on runtime dispatch with string-keyed registries.

## 4. DAG / Execution Graph [A2, A9, A10]

### Graph model

Temporal does **not use an explicit DAG representation**. Workflow execution is an async Rust future running in a `tokio::task::LocalSet`. The "graph" emerges from the control flow of the user's async function — `execute_activity`, `timer`, `execute_child_workflow`, `execute_nexus_operation` are all futures created via `WorkflowContext` methods.

The SDK enforces determinism via **state machines** — one per command type. When a workflow is replayed, these state machines consume the history and resolve the correct futures in the correct order. The `WorkflowMachines` aggregate (`crates/sdk-core/src/worker/workflow/machines/workflow_machines.rs`) owns all active machines for a single run.

### Port typing

None. All inter-workflow data is serialized as Temporal `Payload` (protobuf bytes). No compile-time port type matching.

### Compile-time checks

The type system enforces input/output types for workflows and activities (via `TemporalSerializable`/`TemporalDeserializable` bounds). The correctness of the workflow graph is enforced at **runtime** via the state machine nondeterminism detection — wrong command sequence causes `NondeterminismError`.

### Comparison with Nebula

Nebula's TypeDAG (L1-L4) provides compile-time port typing, TypeId-level dynamic registration, refinement predicates, and petgraph soundness checks. This is a substantial difference in philosophy: Nebula catches graph errors at compile time; Temporal catches them at runtime during replay.

## 5. Persistence & Recovery [A8, A9]

**Critical design note:** The Temporal SDK has **no persistence layer of its own**. Persistence is owned by **Temporal Server** (a separate Golang service). The SDK is a stateless worker client.

**History model:** Every workflow's state is encoded as a sequence of `HistoryEvent` protobufs stored by the server. The SDK reconstructs workflow state by replaying this history through the state machines on demand (cold start) or by maintaining an in-memory cache (sticky queues optimization — `arch_docs/sticky_queues.md`).

**Recovery semantics:** If a worker crashes, the server detects the missing heartbeat (sticky queue TTL ~5s), times out the sticky task, and requeues it on the normal task queue. Any worker can then pick it up and replay from scratch. This is the core Temporal durability guarantee.

**Workflow run cache:** `crates/sdk-core/src/worker/workflow/run_cache.rs` uses an `LruCache<String, ManagedRun>` keyed by `run_id`. Cache size is configurable (`max_cached_workflows` in `WorkerConfig`). Cache miss → full replay from server history. Cache hit → incremental history delivery.

**Comparison with Nebula:** Nebula owns its persistence (sqlx + PgPool, PgWorkflowRepo, PgExecutionRepo, frontier-based checkpoint, append-only execution log). Temporal shifts all persistence responsibility to the server, which is architecturally cleaner for the SDK but requires operating a full Temporal Server cluster. Nebula's approach enables self-contained deployment modes (desktop mode with SQLite) that Temporal cannot support without a server.

## 6. Credentials / Secrets [A4] — DEEP

### A4.1 Existence

**There is no separate credential layer in the Temporal Rust SDK.** The SDK handles only transport-level authentication (gRPC TLS/mTLS, API key header). There is no credential storage, no credential lifecycle management, and no OAuth2 flow.

**Grep evidence:**
- Searched: `"credential"`, `"secret"`, `"LiveCredential"`, `"CredentialOps"`, `"OAuth"`, `"zeroize"`, `"secrecy"` across all `*.rs` files.
- Found: zero matches for `LiveCredential`, `CredentialOps`, `OAuth`, `zeroize`, `secrecy`.
- Found matches for `"token"` and `"api_key"` only in `crates/client/src/options_structs.rs`.

### A4.2 Storage

No credential storage. API key is stored in memory as `Option<String>` in `ConnectionOptions` (`crates/client/src/options_structs.rs:48`). TLS certificates are stored as `Vec<u8>` (PEM bytes) in `ClientTlsOptions` (`crates/client/src/options_structs.rs:151`). No encryption at rest, no vault integration.

### A4.3 In-memory protection

No `Zeroize`. No `secrecy::Secret<T>`. API key and TLS private key are plain `String`/`Vec<u8>`. The `Debug` impl for `ClientTlsOptions` intentionally omits details (`crates/client/src/options_structs.rs` note: "Intentionally omit details here since they could leak a key if ever printed").

### A4.4 Lifecycle

No CRUD for credentials. API key is set once at connection time. No refresh model. No expiry detection. No revocation.

### A4.5 OAuth2/OIDC

Not present. No OAuth2 flows in the SDK.

### A4.6 Composition

Connection options accept a single `api_key: Option<String>` for all calls on that connection. No per-namespace or per-call credential overrides.

### A4.7 Scope

Credentials are scoped to `Connection` (the gRPC channel). Shared across all namespaces and workflows on that connection.

### A4.8 Type safety

No phantom types for credential kinds. No validated/unvalidated distinction. No compile-time leak prevention beyond Rust's standard borrowing.

### A4.9 vs Nebula

Nebula has: State/Material split, `LiveCredential` with `watch()` for blue-green refresh, `OAuth2Protocol` blanket adapter, `DynAdapter` type erasure. Temporal SDK has: none of the above — only connection-level TLS + API key. This is a significant gap if users need per-activity credential rotation, OAuth2 flows, or multi-provider credential management in workflow logic. Users must build this themselves above the SDK layer.

## 7. Resource Management [A5] — DEEP

### A5.1 Existence

No first-class resource abstraction in the Temporal Rust SDK. DB pools, HTTP clients, caches — all managed by the activity author. The `ActivityContext` provides workflow-execution metadata and a heartbeat API, but no resource lifecycle management.

**Grep evidence:**
- Searched: `"ReloadOutcome"`, `"ResourceScope"`, `"on_credential_refresh"`, `"resource_init"`, `"health_check"` across all `*.rs` files.
- Found: zero matches for any of these.

### A5.2 Scoping

Not applicable. No scope model. Activities get `Arc<Self>` (the implementer struct) which is typically shared across all activity invocations for that worker (`crates/sdk/src/activities.rs:26`: `self: Arc<Self>`). This is effectively worker-level global scope.

### A5.3 Lifecycle hooks

No resource init/shutdown/health-check. Workers have `run()` and the `WorkerInterceptor::on_shutdown()` callback (`crates/sdk/src/interceptors.rs:22`), but this is the worker lifecycle, not individual resource lifecycles.

### A5.4 Reload

No hot-reload, no blue-green, no `ReloadOutcome` equivalent.

### A5.5 Sharing

Resources are shared via `Arc<ActivityImplementer>` captured in the activity invocation closure. Standard Rust ownership — `Arc` for shared ownership. No pooling built in.

### A5.6 Credential deps

Activities cannot declare credential dependencies. No credential-rotation hook.

### A5.7 Backpressure

The `SlotSupplier` trait (`crates/sdk-core/src/worker/mod.rs:1629`) provides **task-level backpressure** for workflow and activity tasks. Two implementations: `FixedSizeSlotSupplier` (semaphore-based) and `ResourceBasedTuner` (adjusts slots based on CPU/memory). This is worker-level concurrency limiting, not resource-pool backpressure.

### A5.8 vs Nebula

Nebula has 4 scope levels, `ReloadOutcome`, generation tracking, `on_credential_refresh` per-resource hook. The Temporal SDK has none of this — resources are entirely the user's responsibility. Nebula is substantially deeper here. Temporal's design is intentional: the SDK is a thin layer, and resource management is "your problem."

## 8. Resilience [A6, A18]

### Retry

**Server-side retries (activities):** `RetryPolicy` proto embedded in `ScheduleActivity` command (`crates/common/protos/local/temporal/sdk/core/workflow_commands/workflow_commands.proto:100`). The server executes retries — the SDK does not retry activities itself.

**Client-side retries (gRPC):** `RetryOptions` (`crates/client/src/retry.rs:35`) uses `ExponentialBackoff` from the `backoff` crate. RETRYABLE_ERROR_CODES = `[DataLoss, Internal, Unknown, ResourceExhausted, Aborted, OutOfRange, Unavailable]` (`crates/client/src/retry.rs:22`). Default: 100ms initial, 1.7x multiplier, 5s max, 10s total, 10 max retries.

**Local activity retries:** `ValidatedRetryPolicy` in `crates/sdk-core/src/retry_logic.rs` implements exponential backoff for local activities only (server-side retries don't apply since local activities run within the workflow task).

### Circuit Breaker / Bulkhead

No circuit breaker. No bulkhead. No hedging. These would need to be implemented at the application level.

### Error classification

`ActivityError` enum (`crates/sdk/src/activities.rs:230`):
- `Retryable { source, explicit_delay }` — server will retry
- `Cancelled { details }` — activity is cancelling
- `NonRetryable(Box<dyn Error>)` — no retry
- `WillCompleteAsync` — external completion

Non-retryable error types can also be declared in `RetryPolicy.non_retryable_error_types` by string match.

### Errors [A18]

`thiserror` is used throughout for typed public errors (`workspace.dependencies.thiserror = "2"` in `Cargo.toml`). `anyhow` is used for internal/test errors. No global `nebula-error`-equivalent crate. Error types are per-crate:
- `crates/client/src/errors.rs` — client-level errors
- `crates/sdk/src/activities.rs:230` — `ActivityError`
- `crates/sdk/src/workflows.rs:119` — `WorkflowError`
- `crates/sdk-core/src/worker/mod.rs` — `PollError`, `CompleteWfError`, etc.

## 9. Expression / Data Routing [A7]

No expression engine. No DSL for data routing between nodes. Workflow data flows through standard Rust async code — activity return values are `await`-ed directly. No `$nodes.foo.result.email` syntax. No sandboxed eval.

**Grep evidence:** Searched for `"expression"`, `"jsonpath"`, `"eval"`, `"sandbox"`, `"template"`, `"dollar_sign"`, `"$nodes"` — zero matches.

This is not a gap in Temporal's design; Temporal workflows are Turing-complete code, so there is no need for a constrained expression DSL. The DSL is Rust itself. Compare Nebula which targets a no-code/low-code audience and needs an expression engine for node connectivity.

## 11. Trigger / Event Model [A12] — DEEP

### A12.1 Trigger types

**Schedule:** `crates/client/src/schedules.rs` implements Temporal Schedules — cron-based (Quartz-style with timezone support), interval, and one-shot (`ScheduleSpec`/`ScheduleAction` at `crates/client/src/schedules.rs:34`). A `StartWorkflow` action fires on each trigger interval. Pausing, triggering immediately, and backfill are supported.

**Signals (external events):** Workflows receive `#[signal]` handlers dispatched from external clients or other workflows. Signals are push-based, delivered by the server as part of a `WorkflowActivation` (activation job `WorkflowActivationJob`).

**Updates (transactional signals):** `#[update]` handlers are the synchronous request-response pattern. Added in Temporal SDK 0.3. Both sync and async update handlers exist (`ExecutableSyncUpdate` / `ExecutableAsyncUpdate` at `crates/sdk/src/workflows.rs:306`, `360`).

**Queries:** `#[query]` handlers are synchronous read-only operations. No state mutation.

**Webhooks:** Not a concept in the SDK. Temporal does not have an HTTP webhook ingress built into the worker. Users would implement an HTTP server activity that fires signals/updates.

**External event sources (Kafka, NATS, etc.):** Not built-in. Users build "activity workers" that consume from a broker and signal workflows.

**DB change / CDC / FS watch:** Not built-in.

### A12.2 Webhook

Not present. The SDK does not allocate HTTP endpoints.

### A12.3 Schedule

Schedule spec via `ScheduleSpec` in `crates/client/src/schedules.rs`. Fields include `intervals`, `cron_expressions`, `calendars` (Quartz-compatible), `jitter`, `timezone_name`. DST and timezone supported (Temporal server handles it). Missed schedule recovery handled by server via `catchup_window` and `overlap_policy` (Skip, BufferOne, BufferAll, CancelOther, TerminateOther, AllowAll). Distributed (no double-fire) because server owns schedule state.

### A12.4 External event

No direct broker integration. Temporal's model: activities are the bridge to external systems. An activity can consume from Kafka and heartbeat or signal. This is by design — Temporal does not own the event transport layer.

### A12.5 Reactive vs polling

**Default model:** Worker polls the server for tasks via long-poll gRPC calls (`PollWorkflowTaskQueue`, `PollActivityTaskQueue`). The server delivers tasks when they are ready (event-driven on the server side, polling on the SDK side). The SDK has no reactive/push delivery; it always polls.

### A12.6 Trigger → workflow dispatch

Schedules dispatch 1:1 to workflow starts. Signals and updates target a specific running workflow instance. No fan-out at the SDK level. Conditional triggers must be implemented in workflow code.

### A12.7 Trigger as Action

**Triggers are NOT actions** in Temporal. Schedules, signals, and updates are separate concepts. Schedules start workflows; signals/updates run workflow code within an existing execution. This is fundamentally different from Nebula's `TriggerAction` abstraction where triggers are typed actions producing typed events.

### A12.8 vs Nebula

Nebula has a 2-stage model: `Source → Event → TriggerAction`. Temporal has: Schedule → StartWorkflow (via server) + Signal/Update → WorkflowActivationJob. The closest analogy is that a Temporal `Schedule` is like a Nebula `ScheduleAction`, and a `Signal` is like a Nebula `EventAction`. But the abstraction levels differ significantly — Nebula's Source trait normalizes heterogeneous inbound data into typed Events; Temporal uses raw protobuf Payloads throughout and delegates normalization to user code.

## 12. Multi-tenancy [A14]

**Namespace isolation:** The closest thing to multi-tenancy is Temporal's **Namespace** concept. Each `Client` is bound to a namespace (`ClientOptions::namespace`). Namespaces are managed at the server level, not the SDK level. The SDK enforces namespace in every RPC call.

**RBAC/SSO/SCIM:** No RBAC in the SDK. Authentication is mTLS or API key (Temporal Cloud). Temporal Server/Cloud handles RBAC, SSO, SCIM at the server layer.

**Comparison with Nebula:** Nebula has `nebula-tenant` with three isolation modes (schema/RLS/database), RBAC, planned SSO/SCIM. Temporal shifts all of this to the server layer. The SDK itself is single-tenant.

## 17. Notable Design Decisions

### D1: SDK-Core as a shared native library for all language SDKs

The most distinctive architectural decision: `sdk-core` is compiled to a C ABI and used by TypeScript (via Neon), Python (via PyO3), .NET (via C bridge), and Ruby. This means the core state machine logic is written once in Rust and shared across all SDKs. The trade-off: breaking changes in `sdk-core` must be coordinated across 4+ language SDKs simultaneously, and the C bridge adds complexity (see `sdk-core-c-bridge/`). **For Nebula:** Nebula currently has no multi-language story, which means this pattern is not immediately applicable. If Nebula ever needs Python/TypeScript bindings, the Temporal model (single Rust core + lang bridges) is worth studying.

### D2: `tokio::task::LocalSet` for workflow isolation

Workflows are executed in a `LocalSet` (`crates/sdk/src/lib.rs:604`) to allow `!Send` futures (`Rc<RefCell>` for workflow state, `LocalBoxFuture` for handlers). This means workflow code cannot call `tokio::task::spawn` (which requires `Send`) and must use `spawn_local` instead. The SDK enforces this via the `SdkWakeGuard`/`WakeTracker` nondeterminism detection mechanism (`crates/sdk/src/workflow_executor.rs`), which flags any wake from outside the SDK's known event sources. **For Nebula:** Nebula's frontier scheduler uses `!Send` isolation similarly. This convergence is interesting — both projects independently landed on `!Send` + `LocalSet` for deterministic/isolated workflow execution.

### D3: State machine per command type for determinism enforcement

Each command type (timer, activity, child workflow, update, nexus op) has a dedicated `StateMachine` implementation. Determinism is enforced by requiring that replayed commands match the sequence of events in history. The `#[fsm]` proc macro (`crates/macros/src/fsm_impl.rs`) generates state machine boilerplate. This is a battle-tested design from the original Temporal Java SDK. **For Nebula:** Nebula's checkpoint/replay model could benefit from per-node state machines rather than frontier-based replay.

### D4: Payload type erasure at the gRPC boundary

All workflow/activity I/O crosses the gRPC boundary as `Payload` protobufs. The SDK deserializes at the user-code boundary via `PayloadConverter`. This means the core state machines never see user types — they operate on byte arrays. **Trade-off:** Type safety is pushed to the application layer; the core is universally applicable. **For Nebula:** Nebula maintains stronger type safety through the full execution path (TypeDAG L1-L4). This is a fundamental divergence in philosophy.

### D5: Server-owned retry semantics

Temporal offloads retry execution to the server. The SDK simply declares retry policies. **Benefit:** Retries survive worker crashes. **Limitation:** SDK cannot implement circuit breakers or hedging natively (the server would need to understand those patterns). **For Nebula:** Nebula's nebula-resilience crate owns retry/CB/bulkhead/hedging entirely in the application layer — more flexible for complex policies but requires the engine to be running.

## 18. Known Limitations / Pain Points

1. **No testing utilities** (Issue #1144): TestWorkflowEnvironment, time skipping, mocked activities — all missing. Every test needs a live server. React count: not captured in JSON (no reactions field returned), but 3 open feature requests filed by community in the same week (2025-05-xx).

2. **No activity/workflow interceptors** (Issues #1139, #1140): The existing `WorkerInterceptor` is activation-level, not semantic-level. Community contributor proposed `ActivityInterceptor` design in March 2026 but awaiting maintainer review.

3. **No Nexus handler side** (Issue #1145): The SDK can call Nexus operations from workflows but cannot implement Nexus handlers. This makes it impossible to build cross-namespace operation handlers in pure Rust.

4. **Dependency bloat** (Issue #687): ~800 crates added transitively when including temporal-sdk. `mockall` leaks to non-dev. `opentelemetry-prometheus` pulls in `protobuf 2.x` as a transitive dep. Contributor offered to help fix this in 2024; status as of v0.3.0 is partially addressed (tonic/prost bumped, some optionalization).

5. **SDK status is prerelease**: The README and SDK crate README both state "currently prerelease." The workflow API (`workflow_context`) is flagged as "still very unstable" in `crates/sdk/src/lib.rs:9`. This SDK should not be used in production Rust workflows without accepting breaking changes.

6. **Missing FailureConverter** (`DataConverter.failure_converter` field has `#[allow(dead_code)]` annotation in `crates/common/src/data_converters.rs:16`): Custom failure serialization is not implemented.

## Key Takeaways for Nebula

1. **SdkWakeGuard / nondeterminism detection** (`crates/sdk/src/workflow_executor.rs`) — Temporal's depth-counter waker trick for detecting non-SDK wakes in a LocalSet is a concrete, borrowable technique for Nebula's engine to detect workflow code calling non-deterministic async operations.

2. **ActivityError explicit variants** — Temporal's `Retryable { source, explicit_delay } / Cancelled { details } / NonRetryable / WillCompleteAsync` enum is cleaner than a string-based error classification. Nebula could adopt this pattern in its action error hierarchy.

3. **Server-owned retry durability** — Temporal's insight that retries must survive worker crashes (server enforces them) is the argument for why Nebula's resilience layer needs to persist retry state across engine restarts, not just in-memory.

4. **Durable distributed schedule** (no double-fire via server ownership) — Nebula's schedule design should ensure single-server schedule ownership or CAS-style distributed exclusion, similar to what Temporal achieves by routing schedules through the server.

5. **Worker-level metric taxonomy** — Temporal's ~35 metric names (especially `act_sched_to_start_latency`, `wf_task_replay_latency`, `sticky_cache_hit/miss`) are a mature reference set for Nebula's observability roadmap.
